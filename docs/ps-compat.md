# Photoshop compatibility: COM techniques and per-effect calibration

Deep reference and canonical workflow for Photoshop compatibility, COM observation, PSD corruption rules, and per-effect calibration. Read this before touching layer-style rendering or PSD descriptor I/O.

## COM scripting techniques

Adobe Photoshop 2026 is installed on this machine and is the ground truth for PSD compatibility work. It is COM-scriptable from PowerShell: `(New-Object -ComObject Photoshop.Application).DoJavaScript($jsx)` (the first call launches Photoshop, ~30s).

- To learn how Photoshop encodes a setting, save two PSDs differing in exactly one UI toggle and byte-diff them. This is how the layer-mask link flag (mask flags bit 0 = unlinked) and the "use global light" handling (`uglg` + image resources 1037/1049) in `src/psd/psd_document_io.cpp` were pinned down.
- To check how Photoshop interprets a Patchy-written file, query Action Manager getters, e.g. `executeActionGet` of a layer reference and read `userMaskLinked` / `userMaskEnabled`.
- To compare renders, export Photoshop's flattened view and diff it against `Compositor::flatten_rgb8` of the same file. Gotcha: Photoshop's `doc.saveAs`/`doc.duplicate` fail with a misleading "disk error (-1)" on documents whose smart-object layers ('PlLd'/'SoLd' blocks) reference missing document-global 'lnk2' data — pre-June-2026 Patchy builds produced such files by dropping the global tagged-block section (now preserved; dangling references are stripped on save). For such damaged files, `doc.selection.selectAll(); doc.selection.copy(true)` (merged), paste into a fresh document, flatten, and save a 24-bit BMP from there. Single composite pixels can be probed without exporting via `doc.colorSamplers` (max 4 exist at once — add/read/remove in a loop).
- Script hygiene: set `app.displayDialogs = DialogModes.NO`, only close documents the script opened, and close with `SaveOptions.DONOTSAVECHANGES`.
- **`PhotoshopSaveOptions.maximizeCompatibility = true` is silently overridden by the app-level File Handling preference**, which is set to Never on this machine — the saved PSD then carries a fake all-white merged composite and any `prefer_flat_composite` comparison sees white. Fixture scripts must force the preference to `queryAlways` via the `fileSavePrefs` Action Manager descriptor for the save and restore the original value afterward (see the photoshop-invert.psd generation, July 2026). Also set `app.preferences.rulerUnits = Units.PIXELS` before using coordinate APIs (`selection.select`, `colorSamplers.add`) — they take ruler units, not pixels. Note `colorSamplers` on an unflattened document can read stale values; sample a flattened `doc.duplicate()` instead.
- Small Photoshop-authored regression fixtures are committed under `test-fixtures/psd/` (e.g. `photoshop-unlinked-mask.psd`, `photoshop-global-light-shadow.psd`). Generate new ones with a COM script rather than hand-crafting bytes.
- Saved-channel ground truth lives in `test-fixtures/psd/photoshop-saved-channels.psd`. Photoshop 2026 writes names and display records for every saved alpha/spot plane, but resource 1053 contains identifiers for alpha channels only. Its COM acceptance probe opens the Patchy resave, checks channel count/order/kind/display options and exact 1x1 histogram samples, saves a copy, reopens it, and repeats the checks.
- Beware confounded controls when byte-bisecting: an early conclusion here was wrong because the "control" file lacked layer-record flags bit 3 and went through Photoshop's legacy path.

## Write rules pinned against PS (silent corruption otherwise)

- Layer record flags bit 3 ("Photoshop 5.0 and later") must be written on every layer — without it Photoshop applies legacy semantics and badly misrenders layers that combine an unlinked mask with effects. The layer mask shapes layer effects (shadow/stroke/glow sources) regardless of the link state — the chain toggle affects move behavior only, never rendering. Effect *output* may still spill onto mask-hidden areas unless the "Layer Mask Hides Effects" blending option is on (tagged block 'lmgm', 4 bytes, first byte = bool; modeled as `LayerStyle::layer_mask_hides_effects` and exposed in the layer style dialog's Blending Options page).
- lfx2 effect **blend modes must be written as full stringIDs** ("multiply", "screen", ...) in the 'BlnM' enum — Photoshop 2026 silently reads 4-char codes ('Mltp', 'Scrn') as Normal (pinned by byte-patching probe PSDs; a 16-mode sweep through PS verified every mode Patchy writes). The parser accepts both forms via `blend_mode_from_descriptor_enum`; the writer emits stringIDs (`blend_mode_descriptor_value`).
- The **GrFl (gradient overlay) descriptor is shape-sensitive**: PS resets its blend mode to Normal unless the descriptor mirrors PS's own 14-item layout (`enab, present, showInDialog, Md, Opct, Grad, Angl, Type, Rvrs, Dthr, gs99, Algn, Scl, Ofst`) - other effect descriptors are not shape-sensitive (drop shadow/outer glow blend modes survive with Patchy's leaner layouts). Gradient stop midpoints (`Mdpn`) belong to the destination/right stop and use a piecewise remap around the 50% value. Patchy reads, renders, edits, and writes independent color and transparency midpoints; the first stop's value is preserved but unused. The stored `Intr` smoothness is applied as Photoshop's Classic cubic interpolation after midpoint remapping. `gs99`/`gradientsInterpolationMethod` selects Classic, Perceptual OKLab, or Linear linear-light interpolation. Patchy also models `Dthr`, `Algn`, percentage `Ofst`, scale, reverse, and all five placement styles. Photoshop aligns `Algn=true` geometry to the layer's visible transparency bounds, not the padded PSD channel rectangle. Linear and Reflected gradients span that rectangle along the selected angle. `bevel_examine.psd` pins the 90-degree case against Photoshop's flattened result.
- The **FrFX (Stroke) descriptor has a separate native gradient shape**. Photoshop 2026 COM captures pin solid Stroke at 10 ordered items (`enab, present, showInDialog, Styl, PntT, Md, Opct, Sz, Clr, overprint`) and gradient Stroke at 19 (`enab, present, showInDialog, Styl, PntT, Md, Opct, Sz, Clr, Grad, gradientsInterpolationMethod, Angl, Type, Rvrs, Dthr, Scl, Algn, Ofst, overprint`). The trailing `overprint` bool is modeled (`LayerStroke::overprint`, default false = PS's default, so untouched writer bytes are unchanged), exposed as the Stroke page's Overprint checkbox, and drives the stroke knockout at render time (see the Stroke bullet under Per-effect calibration notes). Gradient mode still carries a black `Clr` placeholder. Its `Grad` object has the Unicode header name `Gradient`; RGB components and color-stop RGB values are plain doubles, and each color stop orders `Clr, Type, Lctn, Mdpn`. The interpolation enum is `gradientInterpolationMethodType/Gcls`, and the centered offset uses percent units. The six confirmed types are Linear, Radial, Angle, Reflected, Diamond, and Shape Burst; the sixth is the stringID `shapeburst` and is Stroke-only (an earlier note here claimed PS normalizes Shape Burst to Linear — refuted July 2026: COM-authored `shapeburst` enums read back and render as shape burst, and PS's own UI writes that spelling, e.g. RTSocketCity title.psd). **Shape Burst render calibration** (COM probes on an L-shaped matte, `photoshop-stroke-shapeburst.psd/bmp`, bands match within +/-1/255; title.psd text cross-checks): the ramp is LINEAR in the stroke band's exact Euclidean distance field with position 0 at the band's OUTER limit and 1 at the inner limit; each side that is a band limit (not the contour) reaches the same +1 px as the coverage band, so Outside/Inside strokes span size+1, and Center strokes span size+2 with one continuous ramp across the contour (solid pixels sit at `band_out + d_in`, others at `band_out + 1 - d_out`). Reverse flips the position; Scale, Angle, Offset, and Align are IGNORED (a Scale-50 probe rendered identically to Scale-100 — title.psd's Silver stroke stores Scl 85 and PS disregards it). Colors and stop opacities sample WITH endpoint smoothing (the two-stop ramp eases through the full `Intr` catmull curve, S(t) = (t + 3t^2 - 2t^3)/2 at smoothness 4096, unlike pinned linear 2-stop overlays) and are SUPERSAMPLED across the pixel footprint — a 1-2-1 tent of the per-center colors reproduces the title.psd silver band's reduced amplitude and the probe tail bytes exactly (invisible on a linear ramp's interior, which is why sharp sampling also matched the probes mid-band). Photoshop anchors the stroke contour (coverage band AND ramp) at the matte's coverage boundary at SUBPIXEL precision, not the alpha>0 shape — on anti-aliased text the alpha>0 shape is one fringe fatter, landing the band ~1 px outward and closing narrow inter-glyph gaps PS leaves clean. Patchy implements this with `stroke_subpixel_distance_fields` (layer_style_mask_ops): the matte supersamples 3x (bilinear; the odd scale keeps coarse centers on fine centers), thresholds at 0.5 in the fine grid, EDTs exactly, and reads back with a +1/3 px compensation that keeps BINARY mattes bit-identical to the pinned pixel-center convention (binary mattes skip the supersampled path entirely). `photoshop-stroke-aa-matte.psd/bmp` (AA ellipse + rotated square with solid strokes, AA ellipse with a Shape Burst stroke) pins it: straight runs match byte-for-byte and curved-edge staircase errors collapse (probe over-tolerance pixels 3.50% -> 2.13%, all remaining on band-limit arcs where the shared linear coverage ramp overfills against PS's rounder arc AA — the same documented corner divergence as binary shapes). A matte with no half-covered pixel falls back to the painted-pixel shape, and flat low-alpha washes far from any solid pixel still stroke in full (photoshop-stroke-partial-alpha). Known remaining divergence on title.psd: pillow-emboss shading over the stroke band and the glyph interiors is covered by the Pillow Emboss calibration bullet above; the band-limit arc AA remains. `psd_generated_stroke_descriptors_match_photoshop_shape` pins the ID forms, field order, object classes, units, and value types. The acceptance probe opened a Patchy-authored radial Stroke (angle 37, reverse on, scale 73, two color stops, three transparency stops), saved it through Photoshop, closed and reopened the copy, and recovered the displayed values and stop counts. Photoshop applied its normal 8-bit alpha quantization, changing the written 42% middle stop to 107/255 (41.960784%).
- Native **Satin (`ChFX`)** has 12 ordered fields: `enab, present, showInDialog, Md, Clr, AntA, Invr, Opct, lagl, Dstn, blur, MpgS`. The Linear `MpgS/ShpC` contour contains the name `$$$/Contours/Defaults/Linear=Linear` and a `Crv ` list with `(0,0)` and `(255,255)` `CrPt` objects. There is no `Nose` field. Patchy writes this native shape with `AntA=false` after an edit and preserves untouched imported custom curves and `AntA` byte-for-byte. Disabled Satin records remain modeled so their settings survive reopening and can be re-enabled. The acceptance probe opened a Patchy-authored combined Satin/Gradient Overlay/gradient Stroke file without warnings, recovered Satin Screen/37%/-23 degrees/9 px/6 px and `Mdpn` 23/77 through Action Manager, saved and reopened it with the same values, then returned it to Patchy. `photoshop-layer-style-4a-roundtrip.psd` pins that return trip; `photoshop-satin-default.psd` pins Photoshop's untouched native default Satin payload.
- The regenerated **ebbl (Bevel & Emboss)** descriptor mirrors PS 2026's native shape: 22 base items ordered `enab, present, showInDialog, hglM, hglC, hglO, sdwM, sdwC, sdwO, bvlT, bvlS, uglg, lagl, Lald, srgR, blur, bvlD, TrnS, antialiasGloss, Sftn, useShape, useTexture`; the Contour sub-option inserts `MpgS, AntA, Inpr` right after `useShape` and the Texture sub-option appends `InvT, Algn, Scl , textureDepth, Ptrn, phase` after `useTexture`. Conditional keys are OMITTED entirely while a sub-option is off, matching PS. `bvlD`'s enum type is `BESs` (the pre-2026 Patchy writer used `BESl`, which PS tolerated). Style/Technique/Soften (`bvlS`/`bvlT`/`Sftn`) parse and round-trip losslessly but the renderer still draws every bevel as a smooth inner bevel and the dialog does not expose them. Enum spellings: `BESl` = InrB/OtrB/Embs/PlEb + stringID `strokeEmboss`; `bvlT` = SfBL/PrBL/Slmt.
- **ShpC contour objects** carry `Nm  ` + a `Crv ` list of `CrPt` points. The Linear default writes name "Linear" with the two-point identity ramp and NO `Cnty` keys; custom curves write `Cnty` on EVERY point (true = smooth, false = corner), PS 2026's canonical forms. Patchy stores exact contour points (`StyleContour`), so edited bevels round-trip imported custom curves losslessly — unlike Satin, which still normalizes to Linear on edit. `build_style_contour_lut` renders them (natural cubic per smooth run, corner points split runs; the Curves 2.0 spline family).
- The regenerated **patternFill (Pattern Overlay)** descriptor uses PS 2026's native 10-item layout: `enab, present, showInDialog, Md, Opct, Ptrn{Nm,Idnt}, Angl, Scl , Algn, phase{Hrzn,Vrtc doubles}`.
- **Pattern data blocks** (`Patt`/`Pat2`/`Pat3` document-global tagged blocks; codec in `src/psd/psd_patterns.{hpp,cpp}`): each pattern is `{u32 length, u32 version=1, u32 image mode, u16 height, u16 width, UnicodeString name (count includes the trailing NUL), PascalString id (no padding), [768-byte table if indexed], virtual memory array list}`, 4-byte padded per pattern. The VMA list is `{u32 version=3, u32 length, rect, u32 max-channels (PS always declares 24)}` followed by max-channels + 2 slots (`u32 written`, then `u32 length, u32 depth, rect, u16 depth, u8 compression, data`). Color channels occupy the first slots; the transparency plane lives in the LAST slot (index max+1). Compression 0 = raw planar rows (what PS 27.8 writes for small tiles and what Patchy writes); the reader also accepts 1 = PackBits with the per-row u16 count table. Imported blocks stay raw-preserved in `unknown_psd_resources` AND decode into `DocumentMetadata::patterns` for rendering; on save Patchy appends one authored `Patt` block holding only referenced patterns whose ids no preserved raw block covers (dedup by the raw blocks' id scan). Photoshop happily reads files where a Patchy `Patt` block coexists with its own.
- **Standalone `.pat` import** (`src/psd/pat_reader.*`) reads Photoshop's big-endian `8BPT` version-1 container. PAT pattern bodies use the same VMA v3 planes as PSD pattern blocks but have no outer per-pattern length or padding; the file header's count and each body's VMA length provide the boundaries. Indexed PAT records have a 772-byte ACT table (768 RGB bytes plus colors-used and transparent-index fields), while PSD `Patt` records carry 768 bytes. Patchy accepts Gray/Indexed/RGB/CMYK, 8/16-bit, raw and PackBits. Bounded unsupported or undecodable items are skipped with warnings; structural damage stops the remaining record scan, retaining any already-decoded prefix with a warning. The optional trailing `8BIMphry` hierarchy descriptor is ignored. Imports are capped at 32 MiB compressed, 8 million pixels per pattern, 16,384 pixels per axis, and 16 million decoded pixels per file, with a separate aggregate plane-expansion budget. The persistent user library is `<settings directory>/patterns/` (`%APPDATA%/Patchy/patterns/` on Windows; `src/ui/pattern_library.*`): PNG plus JSON per entry, valid UTF-8 source Photoshop ids up to 255 bytes preserved when unique (unsafe ids receive a fresh UUID), imports grouped under the PAT filename, and folders editable in Pattern Manager. Layer Style's Pattern Overlay and Bevel Texture pickers present the same folders as expandable tree rows while retaining a flat combo model for the selected value. When an imported id already belongs to different pixels, the sidecar retains that source id so later imports deduplicate the remapped entry. A chosen preset is copied into the active document as `Authored`; later library deletion cannot break that document. If a library tile collides with different document pixels under the same id, the selected copy receives a document-local UUID. Preview-off, cancel, and the pre-undo restore return the document pattern store to its exact original state. Defaults seed once under `patterns/defaultPatternsVersion`, while the manager's explicit Restore command repairs deleted or changed defaults. The manager's preview pans with any mouse button and wheel-zooms about the cursor, outlining the tile under the viewport center (Seamless-Tile-Preview style), and its "Open as Image" button hands the tile to MainWindow, which must DEFER creating the document until the Layer Style dialog closes: `add_document_session` is forbidden under the preview-dialog edit lock (pinned by `ui_layer_style_open_pattern_as_image_defers_until_dialog_closes`). The manager's combined Import command also accepts plain image files (any QImageReader format, EXIF-rotated, alpha kept, any rectangle, capped at 8 Mpx and 30,000 pixels per axis so authored PSD pattern resources remain writable) via `PatternLibrary::import_image`; image imports land ungrouped with a fresh pattern UUID and no pixel dedup. Full-resolution tiles are cached under both a 16-entry and 64 MiB total limit.

Its source URL, SHA-256, copyright notice, and full MIT text are in `test-fixtures/pat/NOTICE.txt`.
- **Pattern overlay render semantics** (probe BMPs pinned in `photoshop-pattern-{overlay,transparent,anchor,scale}.psd/bmp`): "Link with Layer" (`Algn`) anchors the tile grid at the layer's `fxrp` effects reference point (16-byte layer block, two BE doubles, updated by PS when a layer moves — NOT folded into `phase`); unlinked anchors at the document origin. The descriptor `phase` adds on top in both cases. Known divergence: Patchy's move tool does not yet update `fxrp` (`set_layer_effects_reference_point` in core/pattern_resource.hpp exists for the hook), so a linked pattern stays put when its layer is moved inside Patchy; imported files render at PS's recorded anchor correctly. Filtering: 100% is nearest (byte-exact vs PS), magnification interpolates linearly between texels on the INTEGER grid with sample position `(x - anchor)/scale` (byte-exact vs PS at 250%), minification box-averages the source footprint (within mean ~4/255 of PS at 47%; PS's exact minify weights were not identified). Overlay z-order pinned by pairwise probes: pattern under gradient under color — Color Overlay paints LAST (the historical Patchy color-then-gradient order was inverted and is now fixed). Rotation (July 2026, shared PatternTileSampler): the vector-fill probes pinned PS's document-to-tile mapping as R(angle) @ (p - anchor) / scale with R = [[cos, -sin], [sin, cos]] (pattern appears rotated counterclockwise for a positive angle; phase subtracts in document space before the rotation, unscaled), fixing the previously transposed rotation for pattern overlays too. Interior overlays vs SHAPE-LAYER vector strokes (July 2026, fx-sofi probes): Color/Gradient/Pattern Overlay apply to the fill plane only and the vector stroke composites above them (stroke-only shapes keep the overlay on the stroke; drop shadows and the Stroke effect keep the full fill+stroke silhouette); details in docs/vector-tools.md.
- **Bevel Texture referencing an unresolvable pattern is DISABLED by Photoshop** (pinned on tips.psd, July 2026): the file stores `useTexture=true`, but with the pattern id in neither the document's pattern blocks nor PS's presets, PS 2026 shows the Texture sub-option UNCHECKED and renders without it. Patchy mirrors that: the dialog presents the row unchecked (no missing-pattern banner; re-checking it surfaces the live warning), the renderer skips the missing tile, and an untouched style still preserves the raw lfx2 (`ui_layer_style_bevel_texture_missing_pattern_presents_unchecked`).
- **Bevel light model** (COM-calibrated July 2026, `photoshop-bevel-smooth.psd/bmp` + `psd_photoshop_bevel_smooth_fixture_matches_render`; smooth inner bevels at sizes 5/10, altitudes 30/60, contour off/Linear-50/Linear-100 all match within a few /255): the Smooth height field is Satin's exact TENT blur of the matte (not the old triple-box), the surface is `size` pixels deep (per-axis central difference over 2px scales by `0.5 x Depth x size`), and shading is a properly normalized Lambert `L = N dot Light` split as highlight `(L - sin(alt)) / (1 - sin(alt))` and shadow `(sin(alt) - L) / sin(alt)`. This reproduces PS's full-strength shadow edges under a low light and the NON-MONOTONE highlight band under a high light (slopes steeper than 90-altitude tip past the light) — the old `(nz-1) x sin(alt)` ambient model could not exceed ~0.39 highlight where PS reaches `|L_xy| x hglO`. A LINEAR Contour sub-option at Range r multiplies the slope by `100/r` (recovered slope profiles at Range 50 are exactly twice the Range-100 ones, interior included; Range 100 equals contour-off); non-linear curves keep the height-remap model below. Slopes parallel to the light still shade (their L falls below the flat-face `sin(alt)`).
- **Bevel Texture render** (checker probes): PS raises the DARK texels by default (Invert flips; negative depth equals Invert), and shading covers whole texel plateaus, reproduced by box-smoothing the bump plane (radius 1) before the height-field perturbation with gain `kTextureAmplitude = 6` (doubled when the Lambert calibration halved `normal_scale`, keeping the bump's height gradients unchanged).
- **Bevel Style, Technique, and Soften rendering**: the renderer builds one cached continuous height field, then applies style-specific coverage. Smooth blurs the original fractional matte; Chisel Hard uses the exact Euclidean distance profile; Chisel Soft rounds that profile with a narrow blur; Soften applies an additional deterministic blur. Inner/Outer Bevel clip the appropriate side, Emboss covers both sides, Pillow Emboss uses the calibrated model below, and Stroke Emboss uses the composited enabled Stroke masks and shades after the stroke base. The dialog exposes all five `bvlS` styles, all three `bvlT` techniques, and `Sftn`.
- **Pillow Emboss calibration** (COM probes July 2026, `photoshop-pillow-emboss.psd/bmp` + `photoshop-pillow-emboss2.psd/bmp` + `psd_photoshop_pillow_emboss_fixtures_match_render`; depth sweep 1/5/10/25/50/100/200/500/1000 at size 7 plus size 12 and altitude-60 controls, straight-edge profiles fit within ~1-2/255): Photoshop lights the smooth HALF-size tent ramp directly (shading reach is size/2 per side, NOT the folded |2h-1| field, whose crease cancels the central difference exactly where PS's shading peaks) and FLIPS the interior lighting (the valley mirrors the rim; matte binarized at 0.5). The slope factor is `0.5 x Depth x N` with N = the half-size tent peak `max(2, lround(size/2))`, and Depth is FLOORED at 25% — depths 1 through 25 render identically (RTSocketCity title.psd stores Depth 1% and still shades at quarter strength; fitted factors 0.510/1.025/2.02/4.10/10.05/20.97 for depths 25-/50/100/200/500/1000 against predicted 0.5/1/2/4/10/20, the existing 10x ceiling included). Highlights follow the normalized Lambert split (the depth-1000 probe shows its tip-past inversion) but shadows follow the UNNORMALIZED lighting deficit `(sin(alt) - L_raw)/sin(alt)` — linear in slope, saturating early; the probes pin the shadow gain to the highlight's small-signal gain at every depth, and bottom edges mirror top edges exactly, proving the asymmetry follows highlight-vs-shadow, not interior-vs-exterior. `bvlD` "Out " (direction Down) is the calibrated orientation. **Anti-aliased edges composite BOTH sides**: a fringe pixel carries the exterior-signed shading weighted by its backdrop fraction (1 - alpha) AND the flipped interior shading weighted by alpha simultaneously (pillow-ellipse fringe pixels decompose into a highlight and a shadow at once — `photoshop-emboss-styles.psd/bmp`); a hard 0.5 flip instead speckles curved AA contours with wrong-side shading dots (the title.psd curve artifact). **Plain Emboss (`Embs`) is the SAME calibrated model with one global sign and no flip** — its square profiles are the pillow interior's, mirrored across the contour onto both sides (direction Down reads pressed-in: top edges shadow on both sides, bottom edges highlight), with the same half-size ramp, depth floor, and unnormalized shadows; the depth-1 ellipse/square probes match within a few /255 and depth-100 curves within ~25/255 on mid-slope arcs. **Effect order**: bevel shading derives from the LAYER matte, ignores the Stroke effect, and composites OVER the stroke band (the stroke-interplay probes show byte-identical shading alphas painted on top of a 10px solid band), so strokes render before plain bevels; Stroke Emboss still shades after the stroke base. Known residuals: corner arcs at saturated depths (the separable tent's diagonal gradients undershoot PS's, ~1-2 px per corner), the altitude-60 MID-depth highlight (PS renders a tighter symmetric bump with no far tail — its small-slope highlights vanish at high altitude; the altitude-60 depth-1 probe still fits), and vertical-edge shadows measuring ~8% hotter than the light_x projection predicts. Smooth INNER bevels keep the previously pinned full-size symmetric model; whether the depth floor and the unnormalized shadow also apply there was not probed (photoshop-bevel-smooth used depth 100 only).
- **Known divergence, Contour sub-option compositing** (NON-LINEAR curves only; the Linear-curve Range gain is calibrated exactly, see the bevel light model above): Patchy remaps the bevel's normalized height profile through the contour LUT, the model community PSD renderers use. Photoshop's actual compositing differs for non-monotone curves: a Ring/Cone at range 100 renders as a full-band shading inversion with an inner fade (probe `probe-r2*` captures), which no pure height-remap or lighting-remap model reproduced. Patchy's render still produces the expected ridge/groove structures; revisit if the exact model is ever identified.
- The **acceptance probe**: PS 2026 opened a Patchy-authored file (built-in Bricks overlay at Multiply 80%/200% + Ring contour sub at range 75 + Bumps texture at depth 200%) without warnings, returned every value through Action Manager getters including exact contour points and pattern GUIDs, and its resave re-imported identically (`photoshop-pattern-bevel-roundtrip.psd` pins the return trip). Built-in pattern presets carry fixed GUID-shaped ids (`c4a11e00-…`, src/core/pattern_presets.cpp) that PS accepts and re-embeds.
- **Legacy 'lrFX' is a PS 5.x compatibility mirror and is IGNORED whenever lfx2/lmfx exists** (July 2026): merging it filled in categories the lfx2 deliberately disables (a descriptor drop shadow with enab=false parses to nothing, which looks "missing", and the resurrected legacy shadow then rendered where Photoshop shows none). Only layers with NO descriptor effects block read lrFX. Its 'dsdw' record stores blur/intensity/angle/distance as 16.16 FIXED point with a 0-255 opacity byte (28% stores 71); the pre-July-2026 parser read the fixed fields as raw integers one slot early, so a merged legacy shadow could carry a ~7.8-million-pixel distance and abort the flatten on allocation (`psd_lrfx_legacy_drop_shadow_parses_fixed_point`, `psd_lfx2_disabled_effect_suppresses_legacy_lrfx_if_available`).
- **Multi-instance effects ride 'lmfx'**: when a layer stacks several instances of one effect, PS 2015.5+ writes the 'lmfx' tagged block (payload identical to lfx2: u32 version 0, u32 descriptor version 16, `null` root with single keys plus `dropShadowMulti`/`frameFXMulti`/... lists) and a single-instance compatibility lfx2 beside it. Patchy parses lmfx through the lfx2 parser and treats it as authoritative in either block order; edits drop the raw lmfx (a stale one would win in PS and hide the regenerated lfx2). Patchy's own writer keeps emitting multi lists inside lfx2 only, which PS reads (the .asl acceptance probe's applied "Comic Pow" showed `frameFXMulti` through AM getters). Fixture: `photoshop-lmfx-multi-stroke.psd`.
- **Style presets and .asl files** (codec in src/psd/asl_io.*; feature doc docs/style-presets.md). Container pinned against PS 2026's shipped `Abstract Styles.asl`, the live `Styles.psp` palette file (same '8BSL' envelope), and a PS-authored single-style export: u16 version 2, '8BSL', u16 patterns version 3, u32 patterns-section length (an EMPTY section is length 0 with no count field), 'Patt'-block pattern records, u32 style count, then per style a 4-aligned length-prefixed record of two version-16 descriptors: class 'null' {'Nm  ' TEXT (shipped files use the ZString "$$$/key=Display Name" form; user-saved styles are plain), 'Idnt' TEXT GUID} and class 'Styl' {documentMode (empty Objc), 'Lefx' (same shape as the lfx2 root descriptor, class id "Lefx"), optional 'blendOptions'}. An optional trailing '8BIM'+'phry' block carries PS 2026's preset folder hierarchy (ignored, like .pat). The populated **blendOptions** layout contains 'Opct' UntF#Prc, 'Md  ' enum BlnM as a full stringID ("multiply"), stringID-keyed 'fillOpacity' UntF#Prc, and 'Blnd' channel objects. Patchy models Fill Opacity and omits it when 100%; the four-channel Blend If reference layout remains pinned by `test-fixtures/asl/photoshop-style-blend-options.asl`.
- **Fill Opacity**: Photoshop stores nondefault layer Fill as `8BIM` + `iOpa` + u32 length 4 + one value byte and three zero padding bytes; 37% is byte 94 and 100% normally omits the block. Ordinary modes scale base-content coverage. Color Burn, Linear Burn, Color Dodge, Linear Dodge, and Difference use the special 8-bit Fill kernels calibrated from Photoshop COM renders. Effects use master Opacity and remain visible at Fill 0%. Clipped members fade with their base's Fill, adjustment strength is multiplied by Fill, and folder Fill is ignored.
- **Pattern references and vogk coverage are open-time requirements** (July 2026, byte bisection of a rejected 1200x1799 user file, every variant COM open-tested). Photoshop refuses the WHOLE document ("program error") when (a) a `PtFl` fill or `vstk` pattern stroke paint references a pattern id that resolves neither in the file's `Patt`/`Pat2`/`Pat3` blocks nor in PS's own loaded presets (the preset fallback masked this in early tests - fixture GUIDs live in this machine's presets), or (b) a `vogk` keyDescriptorList covers only some of the vmsk subpath groups (mixed live/non-live layers). Patchy therefore collects pattern ids from vector fills and stroke paints alongside layer styles, embeds a 1x1 fully transparent placeholder for any referenced id with no usable tile (never a dangling reference), and writes vogk/vowv only when every subpath group has an emitting origination entry. Details and the read-side heal in [docs/vector-tools.md](vector-tools.md); regression tests `psd_pattern_fill_shape_embeds_patt_block`, `psd_pattern_fill_missing_tile_writes_placeholder`, `psd_partial_vogk_is_omitted_full_vogk_kept`, `psd_damaged_partial_vogk_import_heals_on_resave`.
- **Smart Filter documents have two extra open-time requirements** (pinned July 2026 by byte-level bisection of a 1200x1799 photo PSD Photoshop refused with "Could not open ... because of a program error"; every variant was COM open-tested). When a parseable FEid cache is present, Photoshop eagerly parses the placed layer's embedded lnk2 file and rejects the WHOLE document if (a) the embedded PSD/PSB's merged-composite RLE rows include any odd byte count (the embed parser reads compressed rows in two-byte units; the same file opens fine standalone, and layer channels, FEid planes, and the outer composite all tolerate odd rows - Photoshop itself writes odd rows there), or (b) the smart-object layer carries no 'lyid' layer id block. Without FEid both defects are ignored, which is why plain smart-object round trips never exposed them. Patchy therefore writes every merged composite with even PackBits rows (one literal split per odd row, decode-identical; `make_packbits_row_even`), normalizes stored embeds on save (`even_composite_rows_normalized`, surgical lnk2 element rebuild preserves foreign wrappers), and synthesizes unique 'lyid' blocks for id-less smart-object layers at write time. Regression tests: `psd_composite_rle_rows_are_even_for_photoshop_embeds`, `psd_smart_object_embed_odd_composite_normalized_on_save`, `psd_smart_object_clean_element_normalization_keeps_wrapper`, `psd_smart_object_layers_get_layer_ids_on_save`.
- Native **Blend If** uses the layer-record blending-ranges field, not a tagged block. Photoshop RGB files write 40 bytes: source then destination ranges for Composite Gray, Red, Green, Blue, and an identity transparency pair. Each four-byte range is black-low, black-high, white-low, white-high. Patchy preserves the original payload until a modeled edit, then patches the first 32 bytes and leaves the final identity pair. Fresh identity stays zero-length. Unknown shapes, nonidentity transparency tails, converted non-RGB ranges, and group-boundary records remain raw-only. Adobe's [PSD field specification](https://www.adobe.com/devnet-apps/photoshop/fileformatashtml/) and [Blend If control description](https://helpx.adobe.com/uk/photoshop/using/layer-opacity-blending.html) are the external references; Photoshop 2026/27.8 COM renders are the executable ground truth.

- Photoshop's document-global `FEid`/`FXid` cache blocks associate records with SoLd `placed`, not source `Idnt`. [Adobe's PSD format specification](https://www.adobe.com/devnet-apps/photoshop/fileformatashtml/) defines outer versions 1, 2, and 3; the Photoshop 27.8 captures use version 3. Each record is `u64 bodyLength`, a Pascal `placed` uuid, record version 1, `u64 cacheLength`, cache planes, and an optional shared filter mask. Cache metadata is rect, depth, and max channel count followed by max+2 written/length/data slots. The verified masks are 8-bit raw or PackBits. Photoshop places 0-3 zero alignment bytes after each length-prefixed record inside the declared block payload; the padding is not part of `bodyLength`, and generic tagged-block padding must not be mistaken for it. The reader also recognizes the unambiguous unpadded multi-record shape written by pre-0.20 Patchy and preserves its original payload unchanged.
- Patchy authors the Photoshop 27.8 shape directly. New Gaussian Blur, High Pass, Median, Dust & Scratches, Surface Blur, Unsharp Mask, Motion Blur, Plastic Wrap, Mosaic, Emboss, Box Blur, Radial Blur, and Add Noise entries write canonical `filterFXStyle` objects in the verified key order and share one version-3 FEid record. The FEid cache covers the full document, declares depth 8 and maximum channel 24, puts PackBits RGB in slots 0/1/2 and alpha in slot 25, and stores the shared 8-bit PackBits mask separately. The cached color and alpha are the unfiltered placed Smart Object, not the filtered layer preview. An existing supported descriptor is patched at its known leaves so unmodeled keys and charID/stringID forms remain intact. FEid presence also makes Photoshop eagerly validate the embedded source file and the layer's 'lyid' block; see the even-composite-rows and layer-id write rules above.
- Gaussian rendering was calibrated with Photoshop 27.8 COM one-pixel-line captures at radii 0.1, 0.2, 0.25, 0.3, 0.35, 0.4, 0.45, 0.49, 0.5, 0.51, 0.6, 0.7, 0.8, 0.9, 1, 1.1, 1.5, 2, 2.5, 3, 4, 4.5, and 8 px. Representative normalized byte kernels are radius 0.5 `[55,145,55]`, radius 1 `[2,18,60,96,60,18,2]`, and radius 2.5 `[1,5,11,20,31,39,42,39,31,20,11,5,1]`. The radius-4.5 calibration also retains sub-byte tails measured from the broad step capture; they round to zero for an impulse but accumulate to the native one-level outer halo. The filter is separable and premultiplied-alpha; Photoshop rounds the horizontal result to bytes before the vertical pass. The unfiltered placed raster is embedded in the transparent document-space FEid cache canvas before filtering, so the visible blur can grow past the placed bounds and is alpha-trimmed afterward. Sampling beyond the FEid canvas repeats its nearest edge. Patchy uses the measured table through 8 px and a deterministic recursive approximation above it.
- High Pass uses the same radius semantics but filters straight RGB rather than premultiplied RGBA. Photoshop 27.8 captures pin `clamp(source byte - separately byte-rounded Gaussian byte + 128)` for every RGB channel, unchanged alpha and layer bounds, repeated nearest-edge sampling, and retained hidden RGB at alpha zero. Captures cover 0.1, 1, 4.25, and 10 through 12 px; the radius-10/11/12 impulse kernels are directly pinned, with interpolated practical values between them and the deterministic recursive path above 12 px. A constant RGB field under an alpha ramp becomes neutral 128 at every pixel while its alpha remains byte-identical. Direct, gallery, and native rendering all call this one primitive.
- Median keeps the layer bounds and takes the upper median of each RGBA channel over a square, edge-clamped window. Its effective radius is `floor(Rds)` with a minimum of 1, while the native descriptor retains the exact fractional value from 1 through 500 px. Alpha is filtered independently. For RGB sampling, every zero-alpha source pixel first borrows straight color from the nearest nonzero-alpha source by squared Euclidean distance; equal distances favor the lower row, then the right column. A partially transparent source keeps its own straight RGB. All-transparent input retains its original hidden RGB bytes. The tiled sliding histogram path is deterministic and bounded independently of radius.
- Dust & Scratches keeps the layer bounds and alpha byte-identical. It computes Median's square per-channel RGB candidate with the same nearest-visible straight-RGB extension. For each pixel, it finds the maximum absolute RGB channel difference between the extended source and median candidate, then replaces all three channels with the candidate only when that difference is strictly greater than `Thsh`. A zero-alpha center compares against its extended color, a partially transparent center keeps its own RGB, and an all-transparent input retains its stored hidden RGB.
- Surface Blur rounds its stored radius with `max(1, floor(Rds + 0.5))`, pads by that effective radius, and alpha-trims afterward, so visible bounds can grow by at most the radius. Within the square edge-clamped window, each channel sample `v` around center `c` has weight `max(0, 5 * Thsh - 2 * abs(v - c))`; the weighted quotient rounds to nearest with ties to even. RGB uses Median's nearest-visible straight-RGB extension, while alpha uses the same weighting formula directly. No finite translation support is advertised because RGB extension can select a visible source anywhere in the input.
- Unsharp Mask keeps the input bounds and alpha. Photoshop 27.8 one-pixel-line captures pin its radius-2.5 low-pass kernel as normalized integer weights `[2,4,12,20,30,38,44,38,30,20,12,4,2] / 256`, which differs from Gaussian Blur's radius-2.5 kernel. It computes signed byte detail, truncates `detail * Amnt / 100`, removes `Thsh` from the resulting magnitude, and adds the remainder back to the source. This scale-first ordering is pinned by the 175%, radius 2.5, threshold 7 capture.
- Motion Blur embeds the current result in the transparent document-space FEid canvas, applies one fixed premultiplied-RGBA line from the user-entered angle, and alpha-trims afterward. Distance `d` uses `d + 1` evenly weighted samples; Photoshop 27.8 angle-zero captures pin distance 12 as thirteen 20/255 impulse taps and distance 1 as two 128/255 taps. Direction and bilinear weights are explicitly quantized for cross-toolchain determinism.
- Plastic Wrap uses one deterministic integer height-field renderer: edge-clamped box smoothing, local gradients, and fixed upper-left relief/specular shading. ## Brightness/Contrast legacy calibration (July 2026)

Nine 256-gray-ramp COM captures (Photoshop 2026 legacy-mode adjustment layers at b/c 30/0, 0/25, 0/50, 0/75, 0/100, -40/-40, -50/100, 60/30, 30/-60, flattened to 24-bit BMPs) pin `brightness_contrast_channel_value` in core/adjustment_layer.cpp:

- c = 0: plain add, `clamp(v + b)` — exact in the captures.
- 0 < c < 100: brightness folds into the INPUT and the slope expands around the 127.5 pivot: `clamp(lround((v + b - 127.5) * 100/(100-c) + 127.5))`. Note the slope is `100/(100-c)`, NOT the destructive filter's `(100+c)/100`.
- c = 100: hard threshold, `(v + b) >= 127 ? 255 : 0` — exact, including the b=-50 interaction capture.
- c < 0: contrast compresses around 127.5 FIRST, then brightness adds to the OUTPUT: `clamp(lround((v - 127.5) * (100+c)/100 + 127.5) + b)`.

The hybrid application order (brightness inside for positive contrast, outside for negative) is required by the interaction captures: 60/30 and -50/100 reject brightness-after, -40/-40 and 30/-60 reject brightness-before. Residual: every capture matches within +/-1 (286 of 2,304 samples off by exactly 1 — Photoshop's own LUT construction shows sub-LSB fixed-point noise); the c=0, c=100, and both c=100 interaction captures match exactly. Do not "unify" this formula with the destructive `patchy.filters.brightness_contrast` in either direction: the destructive output is byte-pinned, and the adjustment layer must render Photoshop's math because it round-trips as a native legacy `brit` record.

## Curves 2.0 engine and editor

- `CurvesAdjustment` stores independent ordered control points for Composite RGB, Red, Green, and Blue. A curve has 2 to 19 points; inputs and outputs are 0 to 255, duplicate inputs resolve deterministically, and the first/last input may move inward like Photoshop's black/white input points. Values outside the first/last input clamp to that point's output.
- `build_curve_lut` is the only point-curve interpolation path. Photoshop 2026 uses a natural cubic through the control points with zero second derivative at both endpoints, clamps outside movable endpoints, and rounds to the nearest byte. `build_curves_lut` applies each component table first and then Composite RGB. Destructive Curves, adjustment-layer rendering, thumbnails, and the editor graph consume these core LUTs instead of maintaining separate curve math.
- Calibration used exact 0 through 255 RGB ramps exported from Photoshop as uncompressed 24-bit BMPs. Identity, invert, legacy three-point, nonuniform, nonmonotonic, plateau, one-input transition, moved-endpoint, 16-point alternating, and isolated component/composite cases matched all 3,072 measured bytes exactly. The combined Red plus Composite capture also pins component-first composition in `curves_lut_matches_photoshop_2026_calibration`.
- Old Patchy shadow/midtone/highlight values load as Composite anchors at inputs 0, 128, and 255. Historical `plAD` v4 files may also carry the length-delimited `CRV2` v1 rich-point tail; malformed or future tails fall back to those three anchors, and a v1 payload is bounded to 324 bytes before copying. Editable Curves saves now migrate that modeled result to native `curv` and omit `plAD`, because Photoshop treats a second private adjustment block as unknown data and warns on open. Malformed native `curv` remains on the opaque Pixel path and retains both raw blocks. This deliberately trades old-Patchy editability and preservation of unmodeled private tails for warning-free Photoshop files; all current modeled points are lossless in native `curv`.
- The reusable editor caches a bounded four-channel histogram before painting (at most about 262,000 evenly spaced samples, with fully transparent pixels skipped); any full-resolution composite intermediates are released before the dialog opens. New adjustment layers sample the current document composite. Re-editing an unclipped layer samples the composited layer-tree prefix immediately below it, so Auto is based on the adjustment's input instead of its output. Auto stays disabled for clipped adjustments because their true input exists only inside the compositor's isolated clipping buffer. The graph draws the engine LUT over a four-by-four grid, identity baseline, and channel overlays; supports mouse, numeric, keyboard, and read-only on-canvas point editing; and provides deterministic histogram Auto and full Reset. The black/gray/white image samplers rebuild component curves from their default neutral targets. Reading the canvas never changes the foreground color or history; applying a sample updates only the dialog's normal preview until the user accepts. Image > Curves previews asynchronously but accepts as one destructive pixel undo step. New and edited adjustment layers keep the same point model.
- The preset strip uses eight stable built-in ids and code-generated tonal thumbnails rendered through the same core LUT; a graph edit switches the selection to Custom. `.acv` Load and Save accept Photoshop's version 4 counted RGB shape, legacy version 1 bitmaps, and the indexed `Crv ` extension. Export writes Photoshop 2026's five-curve RGB shape with its trailing identity compatibility curve.
- Native `curv` captures from Photoshop 2026 use a zero map-mode byte, version 1, a big-endian u32 changed-channel bitmap (despite Adobe's table calling it two bytes), implicit point records, and an indexed `Crv ` version 4 extension, padded to four bytes. Channel ids/bits are Composite=0, Red=1, Green=2, Blue=3; points are output then input. Identity has bitmap/count zero and a 20-byte payload. Patchy writes that shape, keeps an untouched imported payload byte-for-byte, and regenerates only after a modeled point edit. A native block is authoritative over `plAD`; malformed native data stays opaque rather than exposing stale controls. Photoshop COM probes open and resave Patchy-written masked and clipped files as editable `LayerKind.CURVES`, and Patchy recovers all points plus mask/clipping from the Photoshop copies. The Photoshop-authored fixtures pin the same round trips in the core suite.
- The momentary Before button restores the same exact original path as disabling Preview without changing the checkbox. Shadow, highlight, and combined clipping views are derived once per canvas-render update into a cached false-color image; painting and zooming reuse it without scanning layer pixels per repaint. Active Red/Green/Blue channels restrict clipping indications to that component.

## Per-effect calibration notes

- **Layer Style dialog state**: footer `Preview` is transient and always opens enabled; turning it off restores the original canvas appearance while edits continue accumulating in the dialog. `Show Effects` on Blending Options is the persisted `masterFXSwitch` (`LayerStyle::effects_visible`). Accept returns the edited settings even if Preview is off. Gradient Overlay and gradient Stroke share the callback-driven stop editor but keep separate host-owned vectors; midpoint diamonds and the numeric Midpoint field edit the destination stop without sorting those working vectors in place. The Satin page exposes blend mode, color, opacity, angle, distance, size, and invert. The Bevel page exposes the modeled highlight/shadow blend mode, color, and opacity fields. Group layer effects are still preservation-only: the compatibility report, import notice, and dialog warn that Preview cannot show a group result.
- **Blend If rendering/editor**: the four active Gray/R/G/B channel gates multiply, and This Layer multiplies Underlying Layer. A split black range `[a,b]` retains byte `v` as `(v-a+1)/(b-a+1)` for `a <= v < b`; joined `a=b` is a hard cutoff. The white side mirrors that inclusive feather and becomes zero immediately above `white_high`. Composite Gray uses Photoshop-calibrated integer weights `299R + 590G + 111B`, rounded at 1/1000. Underlying coverage is `(1-da) + da*gate`, so transparency always passes and partial destination alpha mixes the gate. Layer effects do not inherit the base-pixel gate. Adjustments evaluate This Layer against the adjusted output and Underlying Layer against the pre-adjustment backdrop. A nondefault folder range evaluates its isolated rendered group against the outside backdrop. Blend If on a clipping base does not shrink the original clipping coverage. The dialog exposes channel-tinted ramps, four split-capable handles per This/Underlying row, Alt/Option splitting, numeric values, keyboard editing, Reset Channel, exact Preview/Cancel restoration, and explicit replacement for preview-locked payloads.
- `photoshop-blend-if-4b-roundtrip.psd` is a Photoshop-resaved Patchy file containing distinct normal-layer, native Levels, and folder values; Photoshop retained every modeled byte and wrote the identity boundary record. `photoshop-blend-if-4b-render.bmp` is Photoshop's duplicate-and-flatten result for the same layers. Patchy's composite stays within 2/255 per channel (mean absolute delta below 0.60) across the fixture. Threshold, channel multiplication, destination-alpha, effects, adjustment, group, and clipping behavior were also isolated with 1D COM probes.
- **Satin rendering**: Photoshop derives the effect from the post-mask alpha matte `A`. With `v=(round(max(1,d) cos(angle)), round(-max(1,d) sin(angle)))`, it forms the signed difference between `A(x+v)` and `A(x-v)`, applies Size blur, then takes the absolute value. Size 0 skips blur; positive integer Size `n` uses the exact separable tent kernel `(N-|k|)/N^2` for `|k|<N`, where `N=max(2,n)`. Invert applies `1-q` after the Linear contour. Patchy blends the configured Satin color and mode into the layer's straight RGB before the layer is composited, so source alpha, masks, layer opacity, and the backdrop are each applied once. Custom contour curves and `AntA` are preserved raw while untouched; edited Satin normalizes to the non-anti-aliased Linear contour.
- **Stroke layer effect**: renders as an exact-Euclidean **distance band** from the binary contour ({alpha > 0}), not a morphological dilation: `stroke_alpha_mask` in `src/render/layer_compositor.hpp` computes per-pixel coverage `clamp(a*cov(d_in, band_in) + (1-a)*cov(d_out, band_out))` with `cov(d, band) = clamp(band + 1 - d)` (`kStrokeContourOffset = 1.0`; the contour sits 0.5px past the last pixel center and the AA ramp is 1px). Position bands: Outside = size out, Inside = size in, **Center = size/2 each way** (the old code dilated the full size both ways, drawing Center at double width). The sum combine (not max) keeps the band seamless over anti-aliased contour pixels. Calibrated against Photoshop 2026 COM renders: band runs match PS run-for-run at every position on rects and circles, sizes 5-20 (`test-fixtures/psd/photoshop-stroke-positions.psd` + `psd_photoshop_stroke_positions_fixture_matches` pin this). The EDT helper is `exact_squared_distance_transform` (Felzenszwalb-Huttenlocher, deterministic double envelope math, same no-toolchain-variance rule as brush_dynamics). **Layer mask semantics** (re-probed July 2026; the June 2026 "the mask never reshapes the contour, only attenuates output" model was wrong — its uniform-gray-mask probe cannot distinguish the models, and a fully-opaque layer with an inset black mask rendered no stroke at all, the stroke_test.psd bug): by default the mask reshapes the contour exactly like pixel transparency where it is fully black (shape = {alpha > 0 AND mask > 0}) and the band lands on mask-hidden ground at FULL strength; `photoshop-stroke-masked.psd` + `psd_photoshop_stroke_masked_fixture_matches` pin the runs at all three positions. With "Layer Mask Hides Effects" ('lmgm') the contour ignores the mask and the output is hidden where it lands instead (COM-probed: an opaque rect with its right half masked strokes only the revealed half of the pixel contour; note 'lmgm' rides INSIDE the Lefx descriptor as `layerMaskAsGlobalMask` when set via Action Manager — a bare `setd` on the layer wipes the effects instead). Fractional (gray/feathered) mask values deliberately do NOT fold into the shape or the coverage math: PS composites them with its content-knockout model (a uniform 50% gray mask renders content-at-half plus stroke-filling-the-complement across the WHOLE interior, and an inside stroke on a half-alpha matte replaces the entire shape's content — COM-probed) while Patchy keeps the raw-pixel coverage there, no wash — the same accepted knockout divergence as fills, below. Known limitation: on uniform semi-transparent fills PS composites Inside/Center strokes with a content **knockout** (content is replaced by stroke within its own alpha, e.g. a 50%-alpha rect with an inside stroke renders green-over-background across the whole shape); Patchy over-composites instead, so such interiors show a tinted wash. Exact knockout needs a per-layer compositing buffer. Do not "fix" it by binarizing the inside field at {alpha < 1}: that breaks the PS-exact AA-edge band positions (verified against the circle probes both ways). Opaque content (text, shapes) matches PS essentially exactly, which is the case that matters. **Overprint band knockout** (COM probes July 2026; distinct from the fill-alpha limitation above, which stays divergent): with the FrFX `overprint` bool OFF — Photoshop's default — the stroke band knocks the layer's own content out at FULL band coverage regardless of stroke opacity, and the stroke blends against the layers below with its own blend mode. Pinned semantics: a 29% white inside stroke over gray content on orange renders `0.29 x white + 0.71 x orange` with zero gray in the band; interior effects ride the content plane and vanish under the band too (a 100% Color Overlay probe); clipped members are knocked out with their base; an opaque Multiply stroke multiplies the BACKDROP, not the content, and at reduced opacity the band is `lerp(backdrop, blend(stroke, backdrop), opacity)`; layer opacity scales the whole knocked-out plane. Overprint ON blends the stroke over the layer's own content (the historical Patchy behavior, indistinguishable for opaque Normal strokes). The renderer folds this into a per-pixel base-content factor (`StrokeKnockoutPlane`/`stroke_knockout_factor`, layer_compositor.hpp) applied by the base pass, the interior effects, and the satin/vector-restamp paths: Normal strokes use the compensated divisor `(1-C)/(1-s)` — exact, and a structural no-op for opaque solid strokes, so every pinned opaque fixture keeps its historical bytes (`layer_stroke_opaque_knockout_is_invisible` pins the identity) — while non-Normal strokes use plain `(1-C)` (band interiors exact; the 1 px AA fringe stays approximate, the same accepted class as the band-limit arcs). The knockout applies at **0% stroke opacity** too: the stroke paints nothing but the band still knocks the content out at full coverage, showing the pure backdrop (probe + the fixture's ins0 arms; a real-file size-109 inside stroke at 0% wipes the whole layer interior — do not gate the knockout on `opacity > 0`, only the draw). Fixtures `photoshop-stroke-overprint.psd/bmp` + `psd_photoshop_stroke_overprint_fixture_matches_render` pin eight arms (29%/60% off, 29% on, opaque Multiply off, AA ellipse, layer-opacity 50, 0% off/on); the `layer_stroke_overprint_*`/`layer_stroke_knockout_*` unit tests pin each probe individually.
- **Drop shadow and outer glow share ONE COM-calibrated soft-effect pipeline** (July 2026, `prepare_photoshop_soft_effect_mask` in layer_style_mask_ops.cpp; 'dsdw' probes at sizes 5/10/12/17 x spreads 0/8/18 and the glow probe suites matched byte-for-byte on straight edges): (1) Spread expands the matte by the INTEGER radius `lround(spread% x size)` as a GRAYSCALE dilation (max of alpha x area-sampled-disc coverage, exact Euclidean distances; the earlier binarizing component-strength chamfer band overshoots antialiased text by ~20%; radii above 8 still fall back to it). (2) The remaining size blurs with Satin's exact separable tent kernel `(N-|k|)/N^2`, `N = max(2, lround(size)) - spread_radius` — NOT the old triple-box, which was visibly hotter near the contour on thin shapes (the earlier "matches Photoshop already" note was an artifact of loose comparison bounds). Spread 100 or size 0 means no blur (hard band with 1px AA ramp); sizes 1-2 still blur with the N=2 tent. Never reimplement spread as a post-blur gain: saturating a blur's tail exposes the kernel's support as per-glyph boxes jutting out of the shadow (the qual_rca_pinout.psd spread-100 label-plate bug). `compositor_drop_shadow_full_spread_keeps_rounded_support` pins the rounded support.
- **Effect planes composite with burn-mode alpha FOLDED into the color** (COM-probed July 2026 over a gray backdrop): for LinearBurn and ColorBurn effects, Photoshop scales the effect color toward WHITE by its alpha and blends at full coverage — a 50%-opacity black linearBurn shadow subtracts an absolute 255 x alpha from every channel (out = clamp(b - 255 x mask x opacity)), refuting the plain `lerp(b, blend(b, color), alpha)` model, and colorBurn matches the folded curve exactly. Normal, Multiply, and Darken keep the plain model (probed), and the source-affine modes (Multiply, Screen, LinearDodge) are algebraically identical either way. `composite_effect_color` (layer_compositor.hpp) implements the fold for every effect draw; layer PIXEL compositing keeps the standard model. ColorDodge and other non-affine light modes are unprobed and stay on the plain model. This was the dominant tips.psd title divergence: its 28% linearBurn shadow rendered at less than half of Photoshop's darkening under the lerp model, and with the fold the shadow-only title comparison dropped to 0.1% of pixels off.
- **"Layer Knocks Out Drop Shadow" (DrSh `layerConceals`, default ON)** is modeled (`LayerDropShadow::layer_conceals`) and rendered: the layer's transparency shape multiplies its own drop shadow by `(1 - matte)`, so nothing shadows the backdrop under the shape — regardless of fill opacity (the classic fill-0 no-shadow-inside case), master opacity, or a Stroke Overprint knockout (COM probes July 2026: a knocked-out band and a fill-50/fill-0 interior all show the pure backdrop; with the option off the shadow shows through proportionally). Invisible under fully opaque content, which simply covers the shadow either way — that is why it only surfaced once the stroke knockout landed. Exposed as the Drop Shadow page's checkbox; the writer's pinned 12-item DrSh shape already carried the key (previously hardcoded true, now the stored value, so default bytes are unchanged). Fixture `photoshop-shadow-conceals.psd/bmp` + `psd_photoshop_shadow_conceals_fixture_matches_render`; unit tests `drop_shadow_layer_conceals_hides_shadow_under_shape` and `drop_shadow_conceals_composes_with_stroke_knockout`.
- **Inner-shadow / inner-glow Choke** is the interior mirror of Spread (COM-probed with choke 0/50/100 renders): the inverse matte expands geometrically to choke% x size and only the remaining (1 - choke%) x size is blurred, so choke 100 is a hard Euclidean band exactly `size` deep (Patchy matches the PS profiles within 1/255 on axes and diagonals). The inner glow's **Center** source erodes the matte the same way (choke 100 = hard erosion by the full size). `prepare_layer_style_interior_falloff_mask` (layer_compositor.hpp) implements this; choke 0 keeps the historical interior blur (3 box passes of size/2, reach ~1.5 x size vs Photoshop's ~size) bit for bit, so mid chokes still overreach PS a little — deliberate, choke 0 is pinned. The pre-fix code applied choke as a post-blur gain ((1 - blur) / (1 - choke)), which amplified the box kernel's square support into interior boxes and half-tone dust (and ignored Center-source choke entirely); `compositor_inner_shadow_full_choke_keeps_rounded_interior`, `compositor_inner_glow_full_choke_keeps_rounded_interior`, and `compositor_inner_glow_center_choke_erodes_matte_geometrically` pin the geometric behavior.
- **Outer glow "Softer"** (COM-calibrated July 2026 with square/bar/dot/band probes at sizes 1-40, spreads 0-100, Range 25-100; fixtures `photoshop-outer-glow-range.psd/bmp` and `photoshop-outer-glow.psd/bmp`, pinned by `psd_photoshop_outer_glow_fixtures_match_render`): Photoshop's falloff is (1) a GRAYSCALE dilation of the matte by the INTEGER spread radius `lround(spread% x size)` (max of alpha x area-sampled-disc coverage; the binarizing component-strength chamfer band overshoots AA text by ~20%), (2) Satin's exact separable tent blur with `N = max(2, lround(size)) - spread_radius` (spread 100 = hard band, NO blur; sizes 1-2 still blur with the N=2 tent), then (3) the Quality > Range gain: `mask = min(1, blur x 100/Inpr)`. **Range's UI default is 50, so real files render TWICE the raw blur** — an Action-Manager-authored glow that omits `Inpr` gets 100, which is why naive probes miss it (the original halo-too-hot bug came from a full-strength-at-contour smoothstep instead of this pipeline). `prepare_outer_glow_softer_mask` implements it; `GlwT`/`Inpr` are parsed, modeled (`LayerOuterGlow::technique/range`), exposed in the dialog (Technique/Range), and written (PS 2026 re-opened a Patchy resave and returned GlwT/Inpr/Ckmt/blur/Opct intact via Action Manager). Technique "Precise" keeps the legacy distance-ramp render (approximate, uncalibrated; Range is not applied there). Known residuals: spread radii above 8 use the chamfer band fallback, and hard spread-100 corner arcs differ ~1px from PS's area-sampled disc. Inner glow likely shares the Range gain but stays on its pinned historical path (see Choke above); revisit if inner-glow accuracy comes up.

## Photoshop text model (type layers)

Calibrated July 2026 against Photoshop 2026 with COM-authored probe PSDs (committed as `test-fixtures/psd/photoshop-text-*.psd`; measured via engine-data dumps + ink row bands of PS's own layer rasters). The rules Patchy implements (`kLayerMetadataTextLayoutMode == "photoshop"`, set on import of non-Patchy TySh blocks):

- **Engine units are document pixels.** A 24 pt UI size in a 300 dpi document stores `/FontSize 100` (24 x 300/72) with an identity TySh transform - the transform does NOT carry DPI. UI pt = engine size x transform y-scale x 72/dpi.
- **The TySh transform maps text space to document pixels.** Vertical scale (`hypot(yx, yy)`) multiplies font sizes and leading; the x/y scale ratio is a pure horizontal glyph stretch (free transform of type folds the scale into the matrix and leaves engine values untouched, so xx != yy is common in real files). Never average the axes.
- **Style runs omit properties equal to the ResourceDict normal style sheet** (`/TheNormalStyleSheet` index into `/StyleSheetSet`). A run without `/FontSize` uses the sheet's default (usually 12.0). Falling back to any other value renders wrong sizes (the restaurant-menu bug: title runs omitted FontSize and inherited a sibling run's 7.24564).
- **Leading is per-character; a line's baseline advance = the max effective leading among the ENTERED line's characters.** Fixed leading applies only when `/AutoLeading false`; with AutoLeading true the recorded `/Leading` is stale/informational and the effective value is the paragraph auto-leading fraction (`ParagraphSheet /AutoLeading`, default 1.2) x FontSize, sub-pixel exact (24 pt -> 28.8, no rounding). Leading may be smaller than the em (lines overlap by design).
- **Point text anchors the FIRST baseline at the transform translation (tx, ty) exactly**; justification decides whether tx is the line start (0), middle (center), or end (right) of each line. No leading is applied to the first line.
- **Box text puts the first baseline at box top + OS/2 sTypoAscender x size** (largest run on line 1). Measured baselines at em=100: Arial 71.60, Times 69.50, Verdana 76.00, Courier 61.30 vs sTypoAscender 72.80/69.34/76.47/61.28 - within ~1.2%; capHeight and hhea/winAscent disagree wildly. Leading does not move the first baseline. Patchy reads sTypoAscender via QRawFont's OS/2 table (`typographic_ascent_fraction`).
- **Tracking = FontSize x tracking/1000 px added per inter-glyph gap** (not after the last glyph; 10 H's at 24 pt tracking 200 widen by 9 x 4.8 px). Applied as absolute letter spacing.
- **VerticalScale/HorizontalScale scale glyphs only** - auto leading stays 1.2 x FontSize (unscaled by VerticalScale; e10 probe).

Patchy's serialized run format ("patchy.text.runs") gained v3 for this: double sizes, a leading column that is a number (fixed) or `auto`, a tracking column, and the character panel's horizontal/vertical glyph scales (the SNES box template's 90%-width text and 110%-height cartridge codes; glyph pixel size folds V in, QFont stretch carries H/V, tracking's letter spacing multiplies by H, and the exact-size property stays the FontSize basis for leading). Paragraph runs v3 append the auto-leading fraction. Native Patchy-authored text keeps v1/v2 and Qt-natural layout - the Photoshop model is opt-in per layer via the metadata marker, so Patchy-written PSDs reopened later render exactly as before. On export, fixed-leading runs now write `/AutoLeading false` (PS silently ignores fixed leading otherwise), non-zero `/Tracking`, and non-1 `/HorizontalScale`/`/VerticalScale`.

**Text is rendered UNHINTED** (July 2026): Photoshop never runs TrueType hinting; its antialiased modes (Sharp/Crisp/Strong/Smooth) all filter unhinted outlines with ideal metrics. `configure_text_font_smoothing` therefore maps every antialiased `/AntiAlias` mode to `QFont::PreferNoHinting` (mode 0/None keeps `NoAntialias` + full hinting for the pixel-fit look). The old Sharp/Crisp -> full-hinting mapping fattened stems on small-print-era fonts (tips.psd's 1993 Bitstream Futura MdCn BT) and shifted advances enough to collide adjacent glyphs versus Photoshop's raster.

**Imported type layers keep Photoshop's raster until edited** (July 2026): Photoshop
stores each type layer's rendered pixels in the file and displays them untouched until
the text is actually edited, so a missing font never changes a document's appearance on
open. Patchy now matches this: `should_regenerate_imported_text_preview`
(psd_text_write.cpp) keeps Photoshop-authored rasters (`psd_raster_preview`) even when
the layer carries a big drop shadow or outer glow, and regenerates only when the stored
preview is visibly NOT any style run's declared fill color (baked-in effect pixels that
would corrupt the live outer-effect contour) or when the type block is Patchy-authored
(lossless re-render). It used to regenerate every big-effect text layer through the GDI
path, which substituted missing fonts and reflowed the wrap before any edit (the
"Polymega jump test.psd" Carter One repro). Editing a kept raster still warns before
substituting; CLI automation (`--append-text`) substitutes silently. Pinned by
`psd_import_keeps_clean_foreign_styled_text_raster` (synthetic, both arms),
`psd_polymega_text_keeps_photoshop_raster_if_available`, and the updated
`psd_duke_nukem_mobile_text_style_renders_if_available`.

Black/Heavy faces (DirectWrite weight >= 800, e.g. "Arial-Black") resolve to their FULL face name so the renderer's family+style matcher finds the real face; flattening them to family + bold rendered the Bold face (~15% narrower on the SNES blurb). The bold flag stays set, so a missing Black face still falls back to Bold. The options-bar font COMBO must never be fed such a name raw: `QFont("Arial Black")` walks Windows' substitution chain to TAHOMA (no such family exists in Qt's DirectWrite database -- it is family "Arial", style "Black"), so combo updates go through `text_font_combo_font_for_family`.

**Rotated point-text anchoring**: committed placement pins the TEXT-SPACE anchor -- the justification fraction along the reading axis (line STARTS for left text), first-line side on the stack axis -- never a fixed document corner. The document's visual top-left is the line-END corner under a 90-degree rotation, and pinning it slid the SNES back panel along its reading axis by the line-length delta on every conversion. Both alignment variants follow this rule (`psd_point_text_local_bounds_transform_for_pixels` interpolates in local space; the CS-era document-bounds fallback pins the fractionally-corresponding point of the source ink box).

**Coordinate-space gotcha for scaled BOX text**: runs and the metadata box dims (`patchy.text.box_width/height`, from the raw `/BoxBounds`) live in engine units, but a PSD-frame edit session works in a DOCUMENT-space frame. The render call therefore takes a `layout_scale` that folds the transform's vertical scale into glyph sizes WITHOUT scaling box dims (they are already document-space in that session), and the commit stores the frame dims divided back into raw units so the persisted runs + box + transform stay one coordinate system for re-edits. Before this fold, editing a transform-scaled box layer (the menu's CHICKEN cards, TySh scale ~3.2) rendered raw-size glyphs in a document-size frame.

Committing a transformed point-text layer re-renders CRISP through the (aligned) transform even when the font is substituted -- the session's base raster is already substituted-glyph output, so the resample fallback would only deliver those same glyphs blurry (glyphs at engine size scaled up ~4.5x, the blurry-conversion bug). The first re-edit after a conversion settles placement by a few pixels (the conversion anchors via the PSD-recorded ink box, later sessions via the pure transform); every cycle after that reproduces identical geometry (`ui_restaurant_menu_dishes_commit_matches_photoshop_bands_if_available` pins stability and a mid-alpha sharpness bound).

Known gaps: LeadingType 1 (Japanese top-to-top leading), per-run BaselineShift, and VerticalScale interaction with auto leading under a folded transform are not modeled; box-text RE-edits (after the first conversion) resample through the transform since the crisp re-render path is point-text only; during an edit session the caret/selection geometry still comes from Qt's natural line spacing while the baked preview shows the Photoshop layout, so on multi-line imported text the caret can sit a few pixels off the painted line (the committed result is unaffected).

Calibrated July 2026 against Photoshop 2026 with byte-patched `hue2` probe files (patch the colorize triple, have PS open + flatten + save BMP, fit each pipeline stage by exact interval arithmetic over every gray/color input).

- File encoding: hue2 header = version(=2) u16, colorize u8, pad u8, colorize h/s/l i16 x3, master h/s/l i16 x3. Hue is stored -180..180 (203 degrees = -157; PS also accepts a raw 203). Colorize-off files still carry (0, 25, 0) in the colorize triple. Fresh layers append six default hextant band records plus an undocumented 36-byte trailer of (k*60, 100, 50) triples; Patchy writes that exact template for new layers (`kPhotoshopHueSaturationDefaultTail`) and patch-in-place preserves imported bands/trailer byte-identically on re-save.
- Render pipeline (implemented in `apply_colorize`, src/core/adjustment_layer.cpp): `li = (max+min)>>1`; the lightness slider blends toward white/black and ROUNDS; `base = min(li, 255-li)`; `delta = base * kColorizeSaturationScale[s]` (PS's percent conversion sits ~0.05-0.2% BELOW s/100, hence a per-percent table); `q = li + floor(delta + 0.5)` (round) but `p = li - floor(delta)` (truncate) — the asymmetry reproduces PS's alternating 2l/2l+1 channel sums; `mid = p + floor((q-p) * kColorizeHueInterp[hue]/255 + 0.5)`. The hue interpolant is a measured per-degree table: PS's wheel quantizes adjacent degrees together (pairs collapse roughly every 3.5 degrees), pins values near sector boundaries, and deviates sinusoidally (up to ~5/255) from the ideal hexagon ramp, with the six sectors NOT symmetric - a closed form was not found, so all 360 degrees were probed.
- Accuracy: 99.62% of 1.04M probed pixels byte-exact; residuals are +-1 (worst +-2 on 0.16%) confined to lightness-slider extremes and the li=128/s=100 hairline. Master sliders are ignored by PS while colorize is on (verified). The Hue/Saturation ACTION path ('HStr' via executeAction) renders identically to the file path.
- Committed fixture: `test-fixtures/psd/photoshop-hue-saturation-colorize.psd` (64x48, three gray bands + one orange, colorize 203/52/0 - the generic_bg.psd values - left-half layer mask, maximize compatibility on).

## CMYK import color conversion (lcms2 vs Adobe ACE)

Calibrated July 2026 against Photoshop 2026's own `convertProfile("sRGB IEC61966-2.1", RELATIVECOLORIMETRIC, BPC=true, dither=false)` on a 20-patch flat CMYK probe in U.S. Web Coated (SWOP) v2 (headless COM: fill patches via SolidColor.cmyk, flatten, save PSD + converted BMP). ## Clipping masks

Calibrated July 2026 against `test-fixtures/psd/photoshop-clipping-mask.psd` (semi-transparent base + clipped Multiply layer + clipped Levels adjustment): Photoshop 2026's own flatten matches Patchy's compositor byte-for-byte on it. Semantics implemented: the clipping group composites in isolation starting from the base; members blend against the base's COLOR at full strength (destination alpha 1 - NOT weighted by the base's alpha; the semi-transparent region of the fixture pins this), the base's alpha only masks the ensemble, and the merged group blends into the canvas with the BASE's blend mode and opacity. A hidden base hides its whole group. Known simplifications (documented divergences): a styled base folds its effect pixels into the clip shape and merge, `clbl=false` ("Blend Clipped Layers as Group" off) renders as if true, and clipped runs above an adjustment-layer base render unclipped.

COM gotcha discovered while generating fixtures: PSDs authored headlessly (documents.add + edits + saveAs in one COM script) embed a STALE maximize-compatibility composite (the initial solid-fill state, typically white) - reopening and re-saving does not fix it. Ground-truth fixtures therefore get their composite section replaced with Photoshop's own flatten (duplicate + flatten + BMP export, PackBits-repacked); the layer data stays byte-original. Also set `app.preferences.rulerUnits = Units.PIXELS` before any documents.add/selection math, and never name a JSX top-level variable `name` (the global is read-only "Adobe Photoshop" and assignments silently fail).

## Liquify compatibility boundary

Patchy's manual Liquify workspace writes only the resulting ordinary layer pixels. PSD/PSB output uses the normal channel encoding and contains no Patchy-private Liquify block, so Photoshop sees the rasterized appearance. Patchy does not author or edit Photoshop's native Liquify Smart Filter descriptor yet. Smart Objects must be rasterized before Liquify; imported unsupported Liquify stacks remain byte-preserved and preview-locked. The implementation and legal constraints are in `docs/liquify.md`.

Before adding native support, capture clean-room Photoshop PSD pairs that differ in one manual setting, determine the filterFX and FEid/FXid shapes, and add fail-closed descriptor tests. Never approximate an uncalibrated descriptor or preserve only a supported prefix.
