# Photoshop compatibility: COM techniques and per-effect calibration

Deep reference for PSD-compatibility work. The basics (the COM one-liner, the byte-diff method, the two silent-corruption write rules) also appear in AGENTS.md's "Photoshop compatibility verification" section; read this before touching layer-style rendering or PSD descriptor I/O.

## COM scripting techniques

Adobe Photoshop 2026 is installed on this machine and is the ground truth for PSD compatibility work. It is COM-scriptable from PowerShell: `(New-Object -ComObject Photoshop.Application).DoJavaScript($jsx)` (the first call launches Photoshop, ~30s).

- To learn how Photoshop encodes a setting, save two PSDs differing in exactly one UI toggle and byte-diff them. This is how the layer-mask link flag (mask flags bit 0 = unlinked) and the "use global light" handling (`uglg` + image resources 1037/1049) in `src/psd/psd_document_io.cpp` were pinned down.
- To check how Photoshop interprets a Patchy-written file, query Action Manager getters, e.g. `executeActionGet` of a layer reference and read `userMaskLinked` / `userMaskEnabled`.
- To compare renders, export Photoshop's flattened view and diff it against `Compositor::flatten_rgb8` of the same file. Gotcha: Photoshop's `doc.saveAs`/`doc.duplicate` fail with a misleading "disk error (-1)" on documents whose smart-object layers ('PlLd'/'SoLd' blocks) reference missing document-global 'lnk2' data — pre-June-2026 Patchy builds produced such files by dropping the global tagged-block section (now preserved; dangling references are stripped on save). For such damaged files, `doc.selection.selectAll(); doc.selection.copy(true)` (merged), paste into a fresh document, flatten, and save a 24-bit BMP from there. Single composite pixels can be probed without exporting via `doc.colorSamplers` (max 4 exist at once — add/read/remove in a loop).
- Script hygiene: set `app.displayDialogs = DialogModes.NO`, only close documents the script opened, and close with `SaveOptions.DONOTSAVECHANGES`.
- Small Photoshop-authored regression fixtures are committed under `test-fixtures/psd/` (e.g. `photoshop-unlinked-mask.psd`, `photoshop-global-light-shadow.psd`). Generate new ones with a COM script rather than hand-crafting bytes.
- Saved-channel ground truth lives in `test-fixtures/psd/photoshop-saved-channels.psd`. Photoshop 2026 writes names and display records for every saved alpha/spot plane, but resource 1053 contains identifiers for alpha channels only. Its COM acceptance probe opens the Patchy resave, checks channel count/order/kind/display options and exact 1x1 histogram samples, saves a copy, reopens it, and repeats the checks.
- Beware confounded controls when byte-bisecting: an early conclusion here was wrong because the "control" file lacked layer-record flags bit 3 and went through Photoshop's legacy path.

## Write rules pinned against PS (silent corruption otherwise)

- Layer record flags bit 3 ("Photoshop 5.0 and later") must be written on every layer — without it Photoshop applies legacy semantics and badly misrenders layers that combine an unlinked mask with effects. The layer mask shapes layer effects (shadow/stroke/glow sources) regardless of the link state — the chain toggle affects move behavior only, never rendering. Effect *output* may still spill onto mask-hidden areas unless the "Layer Mask Hides Effects" blending option is on (tagged block 'lmgm', 4 bytes, first byte = bool; modeled as `LayerStyle::layer_mask_hides_effects` and exposed in the layer style dialog's Blending Options page).
- lfx2 effect **blend modes must be written as full stringIDs** ("multiply", "screen", ...) in the 'BlnM' enum — Photoshop 2026 silently reads 4-char codes ('Mltp', 'Scrn') as Normal (pinned by byte-patching probe PSDs; a 16-mode sweep through PS verified every mode Patchy writes). The parser accepts both forms via `blend_mode_from_descriptor_enum`; the writer emits stringIDs (`blend_mode_descriptor_value`).
- The **GrFl (gradient overlay) descriptor is shape-sensitive**: PS resets its blend mode to Normal unless the descriptor mirrors PS's own 14-item layout (`enab, present, showInDialog, Md, Opct, Grad, Angl, Type, Rvrs, Dthr, gs99, Algn, Scl, Ofst`) - other effect descriptors are not shape-sensitive (drop shadow/outer glow blend modes survive with Patchy's leaner layouts). Gradient stop midpoints (`Mdpn`) belong to the destination/right stop and use a piecewise remap around the 50% value. Patchy reads, renders, edits, and writes independent color and transparency midpoints; the first stop's value is preserved but unused. The default 50% path retains Patchy's historical arithmetic and bytes. Known difference: Photoshop applies the descriptor's `Intr=4096` smoothness curve after the midpoint remap, while Patchy retains its existing linear default interpolation.
- The **FrFX (Stroke) descriptor has a separate native gradient shape**. Photoshop 2026 COM captures pin solid Stroke at 10 ordered items (`enab, present, showInDialog, Styl, PntT, Md, Opct, Sz, Clr, overprint`) and gradient Stroke at 19 (`enab, present, showInDialog, Styl, PntT, Md, Opct, Sz, Clr, Grad, gradientsInterpolationMethod, Angl, Type, Rvrs, Dthr, Scl, Algn, Ofst, overprint`). Gradient mode still carries a black `Clr` placeholder. Its `Grad` object has the Unicode header name `Gradient`; RGB components and color-stop RGB values are plain doubles, and each color stop orders `Clr, Type, Lctn, Mdpn`. The interpolation enum is `gradientInterpolationMethodType/Gcls`, and the centered offset uses percent units. The five confirmed types are Linear, Radial, Angle, Reflected, and Diamond; Photoshop normalizes attempted Shape Burst values to Linear. `psd_generated_stroke_descriptors_match_photoshop_shape` pins the ID forms, field order, object classes, units, and value types. The acceptance probe opened a Patchy-authored radial Stroke (angle 37, reverse on, scale 73, two color stops, three transparency stops), saved it through Photoshop, closed and reopened the copy, and recovered the displayed values and stop counts. Photoshop applied its normal 8-bit alpha quantization, changing the written 42% middle stop to 107/255 (41.960784%).
- Native **Satin (`ChFX`)** has 12 ordered fields: `enab, present, showInDialog, Md, Clr, AntA, Invr, Opct, lagl, Dstn, blur, MpgS`. The Linear `MpgS/ShpC` contour contains the name `$$$/Contours/Defaults/Linear=Linear` and a `Crv ` list with `(0,0)` and `(255,255)` `CrPt` objects. There is no `Nose` field. Patchy writes this native shape with `AntA=false` after an edit and preserves untouched imported custom curves and `AntA` byte-for-byte. Disabled Satin records remain modeled so their settings survive reopening and can be re-enabled. The acceptance probe opened a Patchy-authored combined Satin/Gradient Overlay/gradient Stroke file without warnings, recovered Satin Screen/37%/-23 degrees/9 px/6 px and `Mdpn` 23/77 through Action Manager, saved and reopened it with the same values, then returned it to Patchy. `photoshop-layer-style-4a-roundtrip.psd` pins that return trip; `photoshop-satin-default.psd` pins Photoshop's untouched native default Satin payload.

## Curves 2.0 engine and editor

- `CurvesAdjustment` stores independent ordered control points for Composite RGB, Red, Green, and Blue. A curve has 2 to 19 points; inputs and outputs are 0 to 255, duplicate inputs resolve deterministically, and the first/last input may move inward like Photoshop's black/white input points. Values outside the first/last input clamp to that point's output.
- `build_curve_lut` is the only point-curve interpolation path. Photoshop 2026 uses a natural cubic through the control points with zero second derivative at both endpoints, clamps outside movable endpoints, and rounds to the nearest byte. `build_curves_lut` applies each component table first and then Composite RGB. Destructive Curves, adjustment-layer rendering, thumbnails, and the editor graph consume these core LUTs instead of maintaining separate curve math.
- Calibration used exact 0 through 255 RGB ramps exported from Photoshop as uncompressed 24-bit BMPs. Identity, invert, legacy three-point, nonuniform, nonmonotonic, plateau, one-input transition, moved-endpoint, 16-point alternating, and isolated component/composite cases matched all 3,072 measured bytes exactly. The combined Red plus Composite capture also pins component-first composition in `curves_lut_matches_photoshop_2026_calibration`.
- Old Patchy shadow/midtone/highlight values load as Composite anchors at inputs 0, 128, and 255. Historical `plAD` v4 files may also carry the length-delimited `CRV2` v1 rich-point tail; malformed or future tails fall back to those three anchors, and a v1 payload is bounded to 324 bytes before copying. Editable Curves saves now migrate that modeled result to native `curv` and omit `plAD`, because Photoshop treats a second private adjustment block as unknown data and warns on open. Malformed native `curv` remains on the opaque Pixel path and retains both raw blocks. This deliberately trades old-Patchy editability and preservation of unmodeled private tails for warning-free Photoshop files; all current modeled points are lossless in native `curv`.
- The reusable editor caches a bounded four-channel histogram before painting (at most about 262,000 evenly spaced samples, with fully transparent pixels skipped); any full-resolution composite intermediates are released before the dialog opens. New adjustment layers sample the current document composite. Re-editing an unclipped layer samples the composited layer-tree prefix immediately below it, so Auto is based on the adjustment's input instead of its output. Auto stays disabled for clipped adjustments because their true input exists only inside the compositor's isolated clipping buffer. The graph draws the engine LUT over a four-by-four grid, identity baseline, and channel overlays; supports mouse, numeric, keyboard, and read-only on-canvas point editing; and provides deterministic histogram Auto and full Reset. The black/gray/white image samplers rebuild component curves from their default neutral targets. Reading the canvas never changes the foreground color or history; applying a sample updates only the dialog's normal preview until the user accepts. Image > Curves previews asynchronously but accepts as one destructive pixel undo step. New and edited adjustment layers keep the same point model.
- The preset strip uses eight stable built-in ids and code-generated tonal thumbnails rendered through the same core LUT; a graph edit switches the selection to Custom. `.acv` Load and Save accept Photoshop's version 4 counted RGB shape, legacy version 1 bitmaps, and the indexed `Crv ` extension. Export writes Photoshop 2026's five-curve RGB shape with its trailing identity compatibility curve.
- Native `curv` captures from Photoshop 2026 use a zero map-mode byte, version 1, a big-endian u32 changed-channel bitmap (despite Adobe's table calling it two bytes), implicit point records, and an indexed `Crv ` version 4 extension, padded to four bytes. Channel ids/bits are Composite=0, Red=1, Green=2, Blue=3; points are output then input. Identity has bitmap/count zero and a 20-byte payload. Patchy writes that shape, keeps an untouched imported payload byte-for-byte, and regenerates only after a modeled point edit. A native block is authoritative over `plAD`; malformed native data stays opaque rather than exposing stale controls. Photoshop COM probes open and resave Patchy-written masked and clipped files as editable `LayerKind.CURVES`, and Patchy recovers all points plus mask/clipping from the Photoshop copies. The Photoshop-authored fixtures pin the same round trips in the core suite.
- The momentary Before button restores the same exact original path as disabling Preview without changing the checkbox. Shadow, highlight, and combined clipping views are derived once per canvas-render update into a cached false-color image; painting and zooming reuse it without scanning layer pixels per repaint. Active Red/Green/Blue channels restrict clipping indications to that component.

## Per-effect calibration notes

- **Layer Style dialog state**: footer `Preview` is transient and always opens enabled; turning it off restores the original canvas appearance while edits continue accumulating in the dialog. `Show Effects` on Blending Options is the persisted `masterFXSwitch` (`LayerStyle::effects_visible`). Accept returns the edited settings even if Preview is off. Gradient Overlay and gradient Stroke share the callback-driven stop editor but keep separate host-owned vectors; midpoint diamonds and the numeric Midpoint field edit the destination stop without sorting those working vectors in place. The Satin page exposes blend mode, color, opacity, angle, distance, size, and invert. The Bevel page exposes the modeled highlight/shadow blend mode, color, and opacity fields. Group layer effects are still preservation-only: the compatibility report, import notice, and dialog warn that Preview cannot show a group result.
- **Satin rendering**: Photoshop derives the effect from the post-mask alpha matte `A`. With `v=(round(max(1,d) cos(angle)), round(-max(1,d) sin(angle)))`, it forms the signed difference between `A(x+v)` and `A(x-v)`, applies Size blur, then takes the absolute value. Size 0 skips blur; positive integer Size `n` uses the exact separable tent kernel `(N-|k|)/N^2` for `|k|<N`, where `N=max(2,n)`. Invert applies `1-q` after the Linear contour. Patchy blends the configured Satin color and mode into the layer's straight RGB before the layer is composited, so source alpha, masks, layer opacity, and the backdrop are each applied once. Custom contour curves and `AntA` are preserved raw while untouched; edited Satin normalizes to the non-anti-aliased Linear contour.
- **Stroke layer effect**: renders as an exact-Euclidean **distance band** from the binary contour ({alpha > 0}), not a morphological dilation: `stroke_alpha_mask` in `src/render/layer_compositor.hpp` computes per-pixel coverage `clamp(a*cov(d_in, band_in) + (1-a)*cov(d_out, band_out))` with `cov(d, band) = clamp(band + 1 - d)` (`kStrokeContourOffset = 1.0`; the contour sits 0.5px past the last pixel center and the AA ramp is 1px). Position bands: Outside = size out, Inside = size in, **Center = size/2 each way** (the old code dilated the full size both ways, drawing Center at double width). The sum combine (not max) keeps the band seamless over anti-aliased contour pixels. Calibrated against Photoshop 2026 COM renders: band runs match PS run-for-run at every position on rects and circles, sizes 5-20 (`test-fixtures/psd/photoshop-stroke-positions.psd` + `psd_photoshop_stroke_positions_fixture_matches` pin this). The EDT helper is `exact_squared_distance_transform` (Felzenszwalb-Huttenlocher, deterministic double envelope math, same no-toolchain-variance rule as brush_dynamics). Known limitation: on uniform semi-transparent fills PS composites Inside/Center strokes with a content **knockout** (content is replaced by stroke within its own alpha, e.g. a 50%-alpha rect with an inside stroke renders green-over-background across the whole shape); Patchy over-composites instead, so such interiors show a tinted wash. Exact knockout needs a per-layer compositing buffer. Do not "fix" it by binarizing the inside field at {alpha < 1}: that breaks the PS-exact AA-edge band positions (verified against the circle probes both ways). Opaque content (text, shapes) matches PS essentially exactly, which is the case that matters.
- **Drop-shadow Spread** expands the matte geometrically before blurring (COM-probed with spread 0/50/100 renders): solid with rounded Euclidean corners out to spread% x size, then blurred by the remaining (1 - spread%) x size. `prepare_layer_style_soft_mask` implements this via `expand_layer_style_mask_in_place` (chamfer distance, 1px ramp, stroke-band contour convention). Never reimplement spread as a post-blur gain: saturating the box blur's tail exposes the kernel's rectangular support as per-glyph boxes jutting out of the shadow (the qual_rca_pinout.psd spread-100 label-plate bug). Spread 0 keeps the historical triple-box blur profile bit for bit (it matches Photoshop already); `compositor_drop_shadow_full_spread_keeps_rounded_support` pins the rounded support.
- **Inner-shadow / inner-glow Choke** is the interior mirror of Spread (COM-probed with choke 0/50/100 renders): the inverse matte expands geometrically to choke% x size and only the remaining (1 - choke%) x size is blurred, so choke 100 is a hard Euclidean band exactly `size` deep (Patchy matches the PS profiles within 1/255 on axes and diagonals). The inner glow's **Center** source erodes the matte the same way (choke 100 = hard erosion by the full size). `prepare_layer_style_interior_falloff_mask` (layer_compositor.hpp) implements this; choke 0 keeps the historical interior blur (3 box passes of size/2, reach ~1.5 x size vs Photoshop's ~size) bit for bit, so mid chokes still overreach PS a little — deliberate, choke 0 is pinned. The pre-fix code applied choke as a post-blur gain ((1 - blur) / (1 - choke)), which amplified the box kernel's square support into interior boxes and half-tone dust (and ignored Center-source choke entirely); `compositor_inner_shadow_full_choke_keeps_rounded_interior`, `compositor_inner_glow_full_choke_keeps_rounded_interior`, and `compositor_inner_glow_center_choke_erodes_matte_geometrically` pin the geometric behavior.
- **Outer glow**'s spread was already geometric (`distance_falloff_mask`) — but note its smoothstep falloff is heavier near the contour than PS's blur-based falloff (known divergence, unchanged).

## Photoshop text model (type layers)

Calibrated July 2026 against Photoshop 2026 with COM-authored probe PSDs (committed as `test-fixtures/psd/photoshop-text-*.psd`; measured via engine-data dumps + ink row bands of PS's own layer rasters). The rules Patchy implements (`kLayerMetadataTextLayoutMode == "photoshop"`, set on import of non-Patchy TySh blocks):

- **Engine units are document pixels.** A 24 pt UI size in a 300 dpi document stores `/FontSize 100` (24 x 300/72) with an identity TySh transform - the transform does NOT carry DPI. UI pt = engine size x transform y-scale x 72/dpi.
- **The TySh transform maps text space to document pixels.** Vertical scale (`hypot(yx, yy)`) multiplies font sizes and leading; the x/y scale ratio is a pure horizontal glyph stretch (free transform of type folds the scale into the matrix and leaves engine values untouched, so xx != yy is common in real files). Never average the axes.
- **Style runs omit properties equal to the ResourceDict normal style sheet** (`/TheNormalStyleSheet` index into `/StyleSheetSet`). A run without `/FontSize` uses the sheet's default (usually 12.0). Falling back to any other value renders wrong sizes (the restaurant-menu bug: title runs omitted FontSize and inherited a sibling run's 7.24564).
- **Leading is per-character; a line's baseline advance = the max effective leading among the ENTERED line's characters.** Fixed leading applies only when `/AutoLeading false`; with AutoLeading true the recorded `/Leading` is stale/informational and the effective value is the paragraph auto-leading fraction (`ParagraphSheet /AutoLeading`, default 1.2) x FontSize, sub-pixel exact (24 pt -> 28.8, no rounding). Leading may be smaller than the em (lines overlap by design).
- **Point text anchors the FIRST baseline at the transform translation (tx, ty) exactly**; justification decides whether tx is the line start (0), middle (center), or end (right) of each line. No leading is applied to the first line.
- **Box text puts the first baseline at box top + OS/2 sTypoAscender x size** (largest run on line 1). Measured baselines at em=100: Arial 71.60, Times 69.50, Verdana 76.00, Courier 61.30 vs sTypoAscender 72.80/69.34/76.47/61.28 - within ~1.2%; capHeight and hhea/winAscent disagree wildly. Leading does not move the first baseline. Patchy reads sTypoAscender via QRawFont's OS/2 table (`typographic_ascent_fraction`).
- **Tracking = FontSize x tracking/1000 px added per inter-glyph gap** (not after the last glyph; 10 H's at 24 pt tracking 200 widen by 9 x 4.8 px). Applied as absolute letter spacing.
- **VerticalScale/HorizontalScale scale glyphs only** - auto leading stays 1.2 x FontSize (unscaled by VerticalScale; e10 probe).

Patchy's serialized run format ("patchy.text.runs") gained v3 for this: double sizes, a leading column that is a number (fixed) or `auto`, a tracking column, and the character panel's horizontal/vertical glyph scales (the SNES box template's 90%-width text and 110%-height cartridge codes; glyph pixel size folds V in, QFont stretch carries H/V, tracking's letter spacing multiplies by H, and the exact-size property stays the FontSize basis for leading). Paragraph runs v3 append the auto-leading fraction. Native Patchy-authored text keeps v1/v2 and Qt-natural layout - the Photoshop model is opt-in per layer via the metadata marker, so Patchy-written PSDs reopened later render exactly as before. On export, fixed-leading runs now write `/AutoLeading false` (PS silently ignores fixed leading otherwise), non-zero `/Tracking`, and non-1 `/HorizontalScale`/`/VerticalScale`.

Black/Heavy faces (DirectWrite weight >= 800, e.g. "Arial-Black") resolve to their FULL face name so the renderer's family+style matcher finds the real face; flattening them to family + bold rendered the Bold face (~15% narrower on the SNES blurb). The bold flag stays set, so a missing Black face still falls back to Bold. The options-bar font COMBO must never be fed such a name raw: `QFont("Arial Black")` walks Windows' substitution chain to TAHOMA (no such family exists in Qt's DirectWrite database -- it is family "Arial", style "Black"), so combo updates go through `text_font_combo_font_for_family`.

**Rotated point-text anchoring**: committed placement pins the TEXT-SPACE anchor -- the justification fraction along the reading axis (line STARTS for left text), first-line side on the stack axis -- never a fixed document corner. The document's visual top-left is the line-END corner under a 90-degree rotation, and pinning it slid the SNES back panel along its reading axis by the line-length delta on every conversion. Both alignment variants follow this rule (`psd_point_text_local_bounds_transform_for_pixels` interpolates in local space; the CS-era document-bounds fallback pins the fractionally-corresponding point of the source ink box).

**Coordinate-space gotcha for scaled BOX text**: runs and the metadata box dims (`patchy.text.box_width/height`, from the raw `/BoxBounds`) live in engine units, but a PSD-frame edit session works in a DOCUMENT-space frame. The render call therefore takes a `layout_scale` that folds the transform's vertical scale into glyph sizes WITHOUT scaling box dims (they are already document-space in that session), and the commit stores the frame dims divided back into raw units so the persisted runs + box + transform stay one coordinate system for re-edits. Before this fold, editing a transform-scaled box layer (the menu's CHICKEN cards, TySh scale ~3.2) rendered raw-size glyphs in a document-size frame.

Committing a transformed point-text layer re-renders CRISP through the (aligned) transform even when the font is substituted -- the session's base raster is already substituted-glyph output, so the resample fallback would only deliver those same glyphs blurry (glyphs at engine size scaled up ~4.5x, the blurry-conversion bug). The first re-edit after a conversion settles placement by a few pixels (the conversion anchors via the PSD-recorded ink box, later sessions via the pure transform); every cycle after that reproduces identical geometry (`ui_restaurant_menu_dishes_commit_matches_photoshop_bands_if_available` pins stability and a mid-alpha sharpness bound).

Known gaps: LeadingType 1 (Japanese top-to-top leading), per-run BaselineShift, and VerticalScale interaction with auto leading under a folded transform are not modeled; box-text RE-edits (after the first conversion) resample through the transform since the crisp re-render path is point-text only; during an edit session the caret/selection geometry still comes from Qt's natural line spacing while the baked preview shows the Photoshop layout, so on multi-line imported text the caret can sit a few pixels off the painted line (the committed result is unaffected).

Calibrated July 2026 against Photoshop 2026 with byte-patched `hue2` probe files (patch the colorize triple, have PS open + flatten + save BMP, fit each pipeline stage by exact interval arithmetic over every gray/color input).

- File encoding: hue2 header = version(=2) u16, colorize u8, pad u8, colorize h/s/l i16 x3, master h/s/l i16 x3. Hue is stored -180..180 (203 degrees = -157; PS also accepts a raw 203). Colorize-off files still carry (0, 25, 0) in the colorize triple. Fresh layers append six default hextant band records plus an undocumented 36-byte trailer of (k*60, 100, 50) triples; Patchy writes that exact template for new layers (`kPhotoshopHueSaturationDefaultTail`) and patch-in-place preserves imported bands/trailer byte-identically on re-save.
- Render pipeline (implemented in `apply_colorize`, src/core/adjustment_layer.cpp): `li = (max+min)>>1`; the lightness slider blends toward white/black and ROUNDS; `base = min(li, 255-li)`; `delta = base * kColorizeSaturationScale[s]` (PS's percent conversion sits ~0.05-0.2% BELOW s/100, hence a per-percent table); `q = li + floor(delta + 0.5)` (round) but `p = li - floor(delta)` (truncate) — the asymmetry reproduces PS's alternating 2l/2l+1 channel sums; `mid = p + floor((q-p) * kColorizeHueInterp[hue]/255 + 0.5)`. The hue interpolant is a measured per-degree table: PS's wheel quantizes adjacent degrees together (pairs collapse roughly every 3.5 degrees), pins values near sector boundaries, and deviates sinusoidally (up to ~5/255) from the ideal hexagon ramp, with the six sectors NOT symmetric - a closed form was not found, so all 360 degrees were probed.
- Accuracy: 99.62% of 1.04M probed pixels byte-exact; residuals are +-1 (worst +-2 on 0.16%) confined to lightness-slider extremes and the li=128/s=100 hairline. Master sliders are ignored by PS while colorize is on (verified). The Hue/Saturation ACTION path ('HStr' via executeAction) renders identically to the file path.
- Committed fixture: `test-fixtures/psd/photoshop-hue-saturation-colorize.psd` (64x48, three gray bands + one orange, colorize 203/52/0 - the generic_bg.psd values - left-half layer mask, maximize compatibility on).

## CMYK import color conversion (lcms2 vs Adobe ACE)

Calibrated July 2026 against Photoshop 2026's own `convertProfile("sRGB IEC61966-2.1", RELATIVECOLORIMETRIC, BPC=true, dither=false)` on a 20-patch flat CMYK probe in U.S. Web Coated (SWOP) v2 (headless COM: fill patches via SolidColor.cmyk, flatten, save PSD + converted BMP). Patchy's import (vendored lcms2 core, `TYPE_CMYK_8_REV -> TYPE_RGB_8`, `INTENT_RELATIVE_COLORIMETRIC`, `cmsFLAGS_BLACKPOINTCOMPENSATION | cmsFLAGS_NOCACHE`) matched 14/20 patches byte-exactly; max per-channel delta 2 (at C78 M19 Y100 K5), mean abs delta 0.133. The residue is lcms2-vs-ACE CLUT interpolation and is not fixable without Adobe's CMM. `dither=false` matters when reproducing this: PS's default 8-bit conversion dither adds per-pixel noise. Useful pins: SWOP K100 -> (35,31,32), C0 M100 Y100 K0 -> (237,28,36) ("Photoshop red"), paper -> white, C100 M100 Y100 K100 -> (0,0,0).

## Clipping masks

Calibrated July 2026 against `test-fixtures/psd/photoshop-clipping-mask.psd` (semi-transparent base + clipped Multiply layer + clipped Levels adjustment): Photoshop 2026's own flatten matches Patchy's compositor byte-for-byte on it. Semantics implemented: the clipping group composites in isolation starting from the base; members blend against the base's COLOR at full strength (destination alpha 1 - NOT weighted by the base's alpha; the semi-transparent region of the fixture pins this), the base's alpha only masks the ensemble, and the merged group blends into the canvas with the BASE's blend mode and opacity. A hidden base hides its whole group. Known simplifications (documented divergences): a styled base folds its effect pixels into the clip shape and merge, `clbl=false` ("Blend Clipped Layers as Group" off) renders as if true, and clipped runs above an adjustment-layer base render unclipped.

COM gotcha discovered while generating fixtures: PSDs authored headlessly (documents.add + edits + saveAs in one COM script) embed a STALE maximize-compatibility composite (the initial solid-fill state, typically white) - reopening and re-saving does not fix it. Ground-truth fixtures therefore get their composite section replaced with Photoshop's own flatten (duplicate + flatten + BMP export, PackBits-repacked); the layer data stays byte-original. Also set `app.preferences.rulerUnits = Units.PIXELS` before any documents.add/selection math, and never name a JSX top-level variable `name` (the global is read-only "Adobe Photoshop" and assignments silently fail).

## Scratch tools linking Patchy's release libs

Quick scratch tools that link Patchy's release libs (e.g. to flatten a PSD through Patchy's reader outside the test suite) compile with:

```powershell
cmd /s /c '"C:\Program Files\Microsoft Visual Studio\18\Community\Common7\Tools\VsDevCmd.bat" -arch=x64 -host_arch=x64 >nul && cl /nologo /std:c++20 /EHsc /O2 /MD /I <repo>\src <tool>.cpp /link /LIBPATH:<repo>\build\release patchy_psd.lib patchy_render.lib patchy_core.lib patchy_color.lib patchy_filters.lib gdi32.lib user32.lib advapi32.lib dwrite.lib'
```

`/MD` is required to match the libs; `advapi32`/`dwrite` are needed by `psd_document_io`'s font-resolution code.
