## Build, test, and release handoff

For code changes, ALWAYS finish work in this repository by refreshing the local release build — `build\release\patchy.exe` must be freshly built from the final working tree at handoff, never stale. If the change is documentation-only or otherwise cannot affect compiled/runtime behavior (for example README text/images, comments, or agent instructions), do not run the full release build/test handoff just for that change; instead report that the release handoff was skipped because the change is non-code.

Required release handoff steps:

1. Build the release preset:

   ```powershell
   cmd /s /c '"C:\Program Files\Microsoft Visual Studio\18\Community\Common7\Tools\VsDevCmd.bat" -arch=x64 -host_arch=x64 >nul && "C:\Program Files\Microsoft Visual Studio\18\Community\Common7\IDE\CommonExtensions\Microsoft\CMake\CMake\bin\cmake.exe" --build --preset release'
   ```

   `build\release\CMakeCache.txt` carries hand-edited flags the preset does not set: `CMAKE_CXX_FLAGS_RELEASE=/O2 /Ob2 /DNDEBUG /Zi` and `CMAKE_EXE_LINKER_FLAGS_RELEASE=/DEBUG:FULL /INCREMENTAL:NO`. They make release builds emit `patchy.pdb`, which is what symbolizes user crash dumps (WER minidumps land in `%LOCALAPPDATA%\CrashDumps`). If you reconfigure from scratch, pass them again with `-D...`; to symbolize a dump from an older build, rebuild that commit in a temp `git worktree` with the same flags (full links reproduce the binary layout) and read the dump against that PDB.

   A running `build\release\patchy.exe` will lock the link step (`LNK1104`); ask Seth to close it rather than force-killing it (he may have unsaved work).

2. Run the release test binaries from `build\release`:

   ```powershell
   .\patchy_core_tests.exe
   $env:QT_QPA_PLATFORM='offscreen'; .\patchy_ui_visual_tests.exe
   ```

3. In the final response, explicitly report whether the release executable exists at:

   ```text
   build\release\patchy.exe
   ```

4. If the change touched platform-guarded code, CMakeLists/presets, or packaging, additionally run the affected remote build(s) best-effort — `scripts\remote\remote-build.ps1 -Target mac` and/or `-Target linux` — and report the result. A mac/linux failure does not block the handoff (Windows is the gate), but it must be reported, not silently skipped.

Do not say a release was created unless the release preset build completed successfully.

## Platform portability (Windows lead; macOS/Linux ports)

Windows is the lead platform and must never regress: the release handoff above stays mandatory for every code change. macOS and Linux build remotely via `scripts\remote\remote-build.ps1 -Target mac|linux` (snapshots the working tree including uncommitted changes — no commits, branches, or index changes — builds on the remote, runs both suites, streams output back). Changes that touch platform-guarded code, CMakeLists/presets, or packaging should run the affected remote build(s); pure Windows-only work need not. Remote-machine details, provisioning, and the full platform-specific site inventory live in [docs/platform.md](docs/platform.md).

- Default to a small, local `#ifdef Q_OS_WIN / Q_OS_MACOS / Q_OS_LINUX` (`_WIN32` in Qt-free code) **with a portable fallback**. Split into a per-OS translation unit (`foo_win.cpp` / `foo_mac.mm` / `foo_linux.cpp` behind `WIN32`/`APPLE`/`UNIX AND NOT APPLE` in CMakeLists, one shared header) only when a site needs Objective-C++/system frameworks or outgrows about a screenful; per-OS files live next to their feature, not in a platform/ directory.
- Window-frame code stays concentrated in `main_window_chrome.cpp`. `MainWindow::use_custom_window_chrome()` is the single gate: true only on Windows (frameless window + custom title-bar controls + Qt-level edge-resize machinery); macOS/Linux use the native frame, and on macOS the default QMenuBar becomes the global menu bar — never call `setNativeMenuBar(false)` outside the gated `configure_window_chrome()`.
- Tests obtain font files only through `tests/test_fonts.hpp` (role-based, per-OS candidate paths; the Windows lists are the exact historical ones so Windows baselines never move). Mac/Linux test-failure triage: fix real bugs > per-platform skip-with-reason > per-platform baseline; never loosen a tolerance globally to make another platform pass.
- File formats must stay byte-identical across platforms: PSD I/O is explicit big-endian fixed-width (`psd_binary.hpp`); keep any new serialization that way (no `size_t`/`long`/`wchar_t` writes, no raw struct dumps).

## Cross-cutting gotchas

- **Argument evaluation order: never read a variable beside `std::move(it)`.** `f(g(x), std::move(x))` is broken: C++ argument evaluation order is unspecified and MSVC goes right-to-left, so `x` is moved into the by-value parameter before `g(x)` reads it. This silently committed empty feathered selections. Compute `g(x)` into a local first, or use an overload that derives the value internally — `CanvasWidget::combine_selection_from_mask(bounds, mask)` exists for exactly this.
- **Reads must not bump layer revisions.** Layer's mutable accessors bump render/content revisions on ACCESS, so read-only code must go through const layers (`std::as_const`) or it silently invalidates every revision-keyed cache. Hunt regressions with `PATCHY_REV_TRACE=1` (stderr REVBUMP lines per accessor+layer).
- **Nothing O(layer pixels) may run per repaint.** Anything reachable from `paintEvent` must be cached or bounded. Use the layer's app-globally-unique content revision for whole-render results, or `pixel_revision()` for results that depend only on its pixel buffer; a full alpha-channel scan per frame made zoom/pan take ~500 ms on a 70 Mpx layer. See [docs/performance.md](docs/performance.md).
- **Cross-toolchain determinism in core algorithms.** RNG is splitmix64 with explicit uniform mapping — NEVER `std::uniform_*_distribution` (implementation-defined output breaks cross-toolchain pixel tests). Geometry/graph code uses integer math or deterministic double envelopes with fixed tie-breaks (brush_dynamics, magnetic_lasso, the EDT in layer styles all follow this).
- **Read folded event modifiers, not `QApplication::keyboardModifiers()`.** The live state can lag the app-level event filter (stuck eyedropper cursor) and offscreen tests never clear it (see Testing notes). Drive cursor/mode changes from the event's modifiers.
- **Append-only enums**: `BlendMode` (core/layer.hpp) and `BrushDynamicControl` ride combo item data, casts, and file maps keyed on the existing order — only append.
- **Never rename persisted identifiers**: hotkey command ids, palette preset ids, stress-test step ids, New Document preset ids, and settings keys are all stored in user settings or files.
- **Byte-stable serialization canaries**: `psd_layered_writer_bytes_are_stable`, `gif_encoder_bytes_are_stable`, and `tool_write_paths_digest_baseline` pin default-path output byte-identically (FNV hashes). Re-pin only for deliberate format/rendering changes, never to make a refactor pass.
- **The CPU compositor is the reference implementation**: any GPU or optimized rendering path must match its golden output (the pinned compositor tests and canaries encode this).
- **Chrome dialogs that react to closing must funnel through `done()`.** `reject()` (chrome X, Esc) hides WITHOUT a QCloseEvent, so closeEvent-based cleanup misses it; but overriding `reject()` to call `close()` is worse: `QDialog::closeEvent` itself calls `reject()` and treats a still-visible dialog as a vetoed close, turning EVERY close path into a no-op (pinned by `ui_tile_preview_window_tracks_document_edits`).
- **New non-modal dialogs must go through `run_non_modal_dialog`** (dialog_utils): it auto-rejects child dialogs when their parent finishes and applies the macOS above-parent native anchoring. Bypassing it reintroduces the buried/orphaned-dialog bugs. `request_patchy_color` allows one picker at a time; transient pickers keep their own position-memory group.
- **QListWidget rows built with `setItemWidget` must paint their own selection** — the opaque row widget hides the `::item:selected` QSS. Give such lists `padding: 0` on `::item` (padding insets the item-widget geometry and bleeds rows), `background: transparent` on the row's children, and keep the global `QCheckBox { border: none; }` (removing it jams labels into checkboxes on mac; offscreen tests cannot catch it).
- **Options-bar controls share one fixed 26px row height** (app stylesheet); a taller control grows the toolbar only while its tool is active and the canvas shifts on tool switch. QStyleSheetStyle inflates QToolButton size hints by +3px, so bar QToolButtons need an explicit QSS min/max-height cap. A free-transform/warp session owns the bar (hides per-tool widgets) to keep the single row.
- **Options-bar numeric ranges go through `configure_toolbar_spinbox`.** It removes the step buttons and installs the shared trailing-chevron popup with a slider and common values. Do not hand-build a plain options-bar spin box or duplicate the popup; dialog and transform-session numeric fields continue to use `configure_dialog_spinbox` when a range popup would not make sense.
- **Application-wide options-bar state must follow the `current_*` mirror pattern** in MainWindow: record the value in a member on the control's signal, apply it at BOTH per-session sites (the new-session setup block and `activate_document_tab`), and hand-update the mirror in `load_tool_settings()` under `QSignalBlocker`. A setter-only-on-current-canvas lambda silently desyncs documents created/activated later (the shape Fill checkbox bug; `ui_shape_fill_and_corner_radius_apply_to_new_documents`).
- **Status bar widgets**: QStatusBar hides every non-permanent widget while a message shows (Patchy almost always shows one), so left-side widgets are manually positioned children of the `ZoomStatusBar` subclass (`src/ui/zoom_status_bar.*`), never `addWidget()`. Blocking refusals (the message is shown INSTEAD of doing what the user asked, or the operation failed) must go through `MainWindow::show_status_error` / `CanvasWidget::report_status_error`, which add a red flash + warning icon; plain `showMessage` stays the path for informational text (confirmations, cancellations, no-op results, readouts). The message string reaches `currentMessage()` unchanged, so tests keep asserting exact text.
- **App-level shortcuts vs canvas keys**: a tool that needs keys an app shortcut owns must accept the ShortcutOverride in `CanvasWidget::event()` while active (the magnetic-lasso Delete/Backspace pattern — see [docs/selection-tools.md](docs/selection-tools.md)).
- **CMake runtime assets are shared copy-once targets** (`patchy_bundled_fonts`, `patchy_qt_runtime`, `patchy_qt_base_translations`). Never attach per-target POST_BUILD copies into the shared output directory — concurrent copies race under parallel Ninja (intermittent release-build failures). New executables call the existing `patchy_copy_*` helpers.
- **Dialog spin boxes that keep their -/+ buttons** must append `dialog_spinbox_button_style()` (src/ui/dialog_utils) to the dialog stylesheet AFTER all children exist (QSS sub-control gotcha documented in that header).
- **Layer context menu: Edit Layer Styles... stays the FIRST item, always** (Seth's standing rule; `ui_layer_context_menu_keeps_edit_styles_on_top` pins it).
- **NEVER SAVE OVER the PSBtest tent/Content files** (`local-test-fixtures/psd/PSBtest/`): `_if_available` tests byte-pin their PS-2023 shapes; pristine originals live in `D:\projects\C2\TableTents\DrinkTable\` (details in [docs/smart-objects.md](docs/smart-objects.md)).

## Legal constraints (patents and licensing)

Binding rules below; the full patent/trademark/copyright record with claim analyses lives in [docs/smart-objects.md](docs/smart-objects.md) "Patents and trademarks", [docs/selection-tools.md](docs/selection-tools.md), [docs/liquify.md](docs/liquify.md), and [docs/file-formats.md](docs/file-formats.md) (HEIF). Read the relevant record before touching any feature named here.

- **Quick Select solves ONCE on mouse-release; do not add live per-mouse-move classification or selection preview before Nov 3, 2029** (Adobe US 8050498). Also deliberate: Enhance Edge is purely geometric (US 8013870), the solve uses ONE window per stroke (US 8121407), and there is no input-driven algorithm auto-switching (US 10698588). `ui_quick_select_stroke_selects_object_and_is_undoable` asserts the no-mid-drag behavior.
- **Magnetic Lasso and brush shape-dynamics patents are all expired**, so live snap DISPLAY is fine there — but the lasso's selection REGION is still built once, in `finish_magnetic_lasso()`, staying clearly outside US 8050498.
- **Smart Objects, Smart Filters, and the filter gallery are patent-cleared (July 2026 sweep)**, subject to the design-around constraints in the next bullet.
- **Surface Blur, Median/Dust & Scratches, and the tilt-shift overlay are deliberate patent design-arounds (July 2026); never regress them.** Surface Blur computes with NO value histogram (US 8594445, to ~2032); Median and Dust & Scratches keep exactly ONE window histogram updated a single pixel value at a time — never per-column, merged, or slid histograms (US 7920741, to 2030); the tilt-shift preview marks band boundaries with short grip bars only, never lines that span the image (Apple US 8971623, to 2032; the ALGORITHM itself is clear). Outputs are pinned bit-identical by the calibrated tests and constraint comments sit at the code sites (smart_filter_renderer.cpp, zoomable_image_preview.cpp).
- **Lens Blur and Iris Blur stay fixed and explicitly geometric (July 2026).** Lens Blur is one user-selected aperture convolution with no depth map, inferred depth, content classification, highlight detection/boost, or per-pixel kernel. Iris Blur computes one fixed round blur and blends it through exactly one explicit ellipse; never add multiple pins, combined blur patterns, Photoshop-style iris boundary widgets, per-pixel variable-radius kernels, bokeh thresholds, learned depth/focal mattes/splatting, or native `blurbTransform` authoring without a new review (US 8831371, US 9019310, US 8824793, US 2025/0104196). Imported Blur Gallery stacks remain byte-preserved and preview-locked; Patchy's Lens/Iris filters are raster-only. See [docs/smart-objects.md](docs/smart-objects.md).
- **Liquify stays manual, generic, and raster-only (July 2026).** The classic brush-warp patent family is expired; the active face-aware family is not. Never add face detection, landmarks, automatic face regions, face-feature sliders, or any deformation controls derived from a detected face without a new legal review (US 9646195). Do not invent or partially rewrite Photoshop's native Liquify Smart Filter descriptor; ordinary layers save the raster result, Smart Objects require rasterization, and imported unsupported stacks remain byte-preserved. See [docs/liquify.md](docs/liquify.md).
- **Plastic Wrap is a fixed local height-field design (July 2026).** Keep its one integer alpha-weighted-luminance, box-smoothing, gradient, and relief formula. Do not add material or lighting inference, object classification, patch search/synthesis, or content-driven algorithm selection without a new patent review. Its PSD compatibility surface is the native `PlsW` three-integer descriptor; its original renderer preserves bounds and alpha but does not claim Photoshop-identical pixels. See [docs/smart-objects.md](docs/smart-objects.md).
- **Classic Unsharp Mask and Motion Blur stay fixed and user-directed.** Unsharp Mask uses one radius-selected low-pass plus explicit Amount and Threshold; it must not rank pixels, infer blur, pick a threshold, search patches, deconvolve, or adapt its kernel from image content. Motion Blur uses one straight line kernel from explicit Angle and Distance; it must not estimate shake, track objects, use optical flow, vary the path per pixel, or grow Blur Gallery-style widgets. Re-read the adaptive-sharpening, deblurring, tracking, and Blur Gallery patent families before adding any of those behaviors.
- **Future-feature patent gates**: NO content-aware fill, patch-based healing, or reshuffle-style synthesis before Adobe's PatchMatch family expires (US 8285055/8340463/8355592, 2031); NO ML/GAN dust-and-scratch removal (US 11393077 family, ~2040; the classic median+threshold filter is outside its claims); re-read US 12530821 (neural compositing) before wiring generative output into smart-filter stacks; never add a one-click "merge visible + apply effects + mask the merged copy" macro without re-reading US 8655112 — smart filters attach to ONE existing layer by design. The classic healing brush is unencumbered (US 6587592 expired 2021).
- **Classic Flow/Airbrush stays a flat-stamp operation.** It may repeat the current 2D brush stamp at fixed spatial and time cadences and cap the accumulated paint at Opacity. Never add particle bursts, a 3D spray cone or stylus-pose projection, velocity-dependent deposition, bristle or fluid-surface simulation, wet/dirty paint, or bidirectional paint transfer without re-reading US 8854342 and the US 9600907/10489937 family. The claim analysis and code site are recorded in [docs/brushes.md](docs/brushes.md).
- **Brush Texture, Color Dynamics, Wet Edges, Dual Brush, and Mixer Brush have binding design boundaries (July 2026; see docs/brushes.md).** Texture stays one static world-anchored generated grayscale mask: never let pressure, velocity, direction, rotation, or stylus pose select/compose/weight texture channels while US 10902645/8896579 are active. Color Dynamics only interpolates the two selected colors plus direct HSV offsets: no canvas sampling, paint mixing, or 3D RGB Bezier path (US 10924633). Wet Edges keeps only one max-coverage union mask for the current stroke, with no canvas-color feedback, paint store, fluid/pigment/paper/drying/pickup simulation. Dual Brush stays one fixed secondary computed mask, never a component graph/hierarchy. Mixer Brush reads exactly one active-layer RGBA pixel at mouse-down and never samples the canvas again during that stroke. Never add repeated pickup, injection/pickup/contamination stores, per-bristle or per-pixel paint state, fill channels, erodible height-map tips, fluid/drying/cleaning simulation, or 3D RGB curves without re-reviewing US 7768525, US 8749572, US 10217253, and US 10924633. ABR input-driven texture depth imports as a static depth with a notice.
- **Pattern Stamp stays a direct static tile lookup.** Never inspect the destination to derive illumination, gradients, shading, or reflections, and never adapt/synthesize the pattern from image content (US 9864922). Never assemble runtime texture channels weighted by pressure, direction, velocity, or orientation (US 10902645). A static pattern sample multiplied by the existing brush-tip coverage is the cleared design; the analysis and code constraint live in [docs/brushes.md](docs/brushes.md).
- **PSD interop stays legally clean by method**: Patchy's ground truth is observed output of a licensed Photoshop, never decompilation. Keep it that way: no verbatim Adobe spec text in the repo or docs (link it instead), self-authored fixtures only, and no support for the undocumented PSDC cloud-document format.
- **Adobe trademark posture (checked May 2026)**: "Smart Object(s)", "Smart Filter(s)", "Filter Gallery", and "PSD" are not Adobe trademarks; PHOTOSHOP is registered and policed. Use referential phrasing ("compatible with Adobe Photoshop"), never put Adobe marks in product names, icons, or branding, and keep README.md's Trademark Notice.
- **Bundled art and textures must never come from AI image generation.** The two acceptable sources are code generation (the brush-tip / pattern-preset model) and real photographs or human-artist work under a commercial-compatible license (CC0 strongly preferred). Every bundled third-party asset gets a NOTICE-THIRD-PARTY.md entry and ships lossless (never re-encode through JPEG). ambientCG: photogrammetry/photo `creationMethod` only; Poly Haven textures are human-made.
- **HEIC/HEIF decode comes from PLATFORM codecs only — never vendor libheif, libde265, x265, or any other HEVC decoder/encoder**; encoding HEIC is off the table entirely. Rationale, patent-pool details, and the per-platform codec design are in [docs/file-formats.md](docs/file-formats.md).
- **Dependency licensing**: commercial-friendly dependencies only in shipping binaries. GPL tools may be used for local development or test generation, but only when they are not linked or distributed.

No Photoshop SDK code or Adobe-created assets are ever allowed in this repository or in any Patchy binary. The only exception is the `test-fixtures/psd/` regression files, which are all authored by Seth and/or the Patchy team, and are not Adobe assets.

## Testing notes

- The core suite is one binary split across `tests/core/*_tests.cpp` (one TU per thematic group, same pure-move rules as the src splits). Each TU ends with a `<group>_tests()` registration function; `tests/core/main.cpp` concatenates them in a fixed order — never add static self-registration (cross-TU init order would reorder the suite). Add a test by appending to the right group TU's registration vector; shared helpers live in `tests/core/core_test_support.{hpp,cpp}` / `psd_test_support.{hpp,cpp}` (`namespace patchy::test`, Qt-free — move helpers there, never copy them).
- The UI suite follows the same design across `tests/ui/*_tests.cpp` (the old tests/ui_visual_tests.cpp monolith is gone): registration declarations in `tests/ui/ui_test_groups.hpp`, concatenated by `tests/ui/main.cpp` in a load-bearing fixed order (the contact-sheet and shot_readme tests depend on artifacts written by earlier tests, and QSettings state leaks between tests by construction). Shared helpers live in `tests/ui/ui_test_support.{hpp,cpp}` (`namespace patchy::test::ui`); `MainWindowTestAccess` (tests/ui/ui_test_access.hpp) is befriended by name in main_window.hpp.
- `patchy_ui_visual_tests.exe` must run with `QT_QPA_PLATFORM=offscreen`. The offscreen platform does **not** enumerate installed Windows fonts; register what a test needs with `QFontDatabase::addApplicationFont("C:/Windows/Fonts/<file>.ttf")`. Never call `removeApplicationFont` to clean up — invalidating an in-use font cache can hard-crash the suite.
- A registered font may not appear under its familiar GDI name: the offscreen FreeType database uses the OpenType *typographic* family (ariblk.ttf = family "Arial" + style "Black"). Patchy resolves such names via `available_text_family_style_match`; tests should not gate on `QFontDatabase::families().contains(...)` checks.
- The test harness `CHECK()` macro throws. A failing CHECK while a `MainWindow` with an open inline text editor is still alive aborts the process during unwind (exit code 3, no `[FAIL]` line printed). Prefer asserting after the editor session is committed/closed.
- Tests save PNGs via `save_widget_artifact(...)` into `test-artifacts/` next to the binary — inspect them to visually confirm rendering behavior.
- The offscreen platform does **not** clear `QApplication::keyboardModifiers()` after a synthetic `QKeyEvent`, and the stuck bit persists across tests in the shared `QApplication`. Assert cursor/mode changes through paths that read the event's *folded* modifiers, not `keyboardModifiers()` (see `ui_brush_alt_shows_eyedropper_cursor` for the order-independent pattern).
- A click on a layer-row mask/content thumbnail can rebuild the layer row (the old row widget is deleted), so never reuse a cached row/thumbnail pointer across the press — use `click_layer_row_thumbnail(...)` in tests/ui/ui_test_support.cpp, which refetches the widget for press and release. Reusing the stale pointer is a use-after-free that flakes only when the heap reuses the freed block.
- If the visual suite dies with an access violation, the log ends with a symbolized stack (a dbghelp vectored handler in `main`) — read it instead of re-running and hoping.
- Run a subset of visual tests by passing a name substring as the first argument (or `PATCHY_UI_TEST_FILTER`): `.\patchy_ui_visual_tests.exe ui_audio_splitter`. There is no `--test` flag.
- Tests that open notice-raising files assert `statusBar()->currentMessage()`; only tests that ENABLE `imports/showPsdWarningsAndInfo` need the REPEATING QTimer dismisser — a one-shot fires during the open-progress phase and the suite hangs (see [docs/file-formats.md](docs/file-formats.md) "Import notices").
- Some tests skip per platform by design (Windows-only chrome/plugins, font metrics); the current list with reasons is in [docs/platform.md](docs/platform.md). Local-fixture (`local-test-fixtures/`) tests `[SKIP]` on the remote machines because that directory is deliberately untracked.

## Driving and inspecting the app

- **Native Windows visual QA uses Patchy's own screenshot command, never Computer Use or other desktop-control automation.** Launch the release app and capture its real native styling with `patchy.exe --screenshot`; inspect the resulting image directly.
- `patchy.exe --screenshot <out.png>` saves a grab of the RUNNING instance's window without raising or focusing it (add `--screenshot-widget <qtObjectName>` and/or `--screenshot-rect x,y,w,h`). The invoking process exits immediately; poll for the output file. Combine with positional files to open a document first. With no instance running, the new process shows the window, captures ~1.5s after startup, and exits (0 saved / 3 failed).
- `patchy.exe --stress-test[=quick|small|standard|huge] [--stress-report-dir <dir>]` runs the built-in performance scene and exits; reports land in `%APPDATA%\Patchy\stress-reports\` (read `stress-latest.json`). Meaningful numbers need a REAL screen; quick (1024 px) for iteration, standard (4096 px) for full-scale. Details in [docs/performance.md](docs/performance.md).
- Debug env vars: `PATCHY_NO_SINGLE_INSTANCE=1` (allow multiple instances), `PATCHY_FAKE_SCANNER_FILE=<path>` (bypass native scanner acquisition for scanner-import tests), `PATCHY_REV_TRACE=1` (log revision bumps), `PATCHY_ZOOM_TRACE=1` (paint/zoom phase timings over 2 ms), `PATCHY_STYLE_MASK_CACHE_OFF=1`, `PATCHY_RENDER_SINGLE_THREADED=1` (byte-stable sequential render for cross-run checksum compares), `PATCHY_PROCESSING_OVERLAY_MIN_PIXELS` (overlay threshold override), `PATCHY_UI_TEST_FILTER` (test subset).
- Composite checksums (stress report FNV-1a and any big-render comparison) are comparable on one machine only: text AA varies across machines and the parallel strip renderer makes them thread-count dependent.

## Photoshop compatibility verification

Adobe Photoshop 2026 is installed on this machine and is the ground truth for PSD compatibility work. It is COM-scriptable from PowerShell: `(New-Object -ComObject Photoshop.Application).DoJavaScript($jsx)` (the first call launches Photoshop, ~30s).

- To learn how Photoshop encodes a setting, save two PSDs differing in exactly one UI toggle and byte-diff them. To check how PS interprets a Patchy-written file, query Action Manager getters (`executeActionGet`) or diff PS's flattened export against `Compositor::flatten_rgb8`.
- Committed PS-authored regression fixtures live under `test-fixtures/psd/`; generate new ones with a COM script rather than hand-crafting bytes.
- Two silent-corruption write rules: layer record flags **bit 3** must be written on every layer (without it PS applies legacy semantics and misrenders unlinked-mask + effects combinations), and lfx2 effect **blend modes must be written as full stringIDs** ("multiply", not 'Mltp') — PS silently reads 4-char codes as Normal. The GrFl (gradient overlay) descriptor is additionally shape-sensitive (must mirror PS's 14-item layout).
- COM techniques, script hygiene, per-effect calibration notes (Stroke band, Spread, Choke, glow falloff), and the scratch-tool compile line are in [docs/ps-compat.md](docs/ps-compat.md).

## Feature index

Read the linked doc before working on a feature; each entry below carries only the constraint most likely to bite.

- **Smart objects and Smart Filters** ([docs/smart-objects.md](docs/smart-objects.md)): a smart object stays a `LayerKind::Pixel` layer identified by `patchy.smart_object.*`; supported native filter stacks rebuild from immutable source data, unsupported stacks stay byte-preserved and preview-locked. E1-E14 rules, PS 27.8 captures, and the patent record live there.
- **Liquify** ([docs/liquify.md](docs/liquify.md)): manual fixed-grid brush deformation is raster-only and patent-cleared; never add face detection, face landmarks, feature-specific face controls, or invented native Smart Filter descriptors.
- **Warp** ([docs/warp.md](docs/warp.md)): all warp features share `generate_style_warp_mesh`/`apply_warp_distortion` (core/warp_mesh); meshes regenerate from parameters, never stored. Box text warps over the dragged FRAME — that IS Photoshop's behavior.
- **Brush tips & dynamics & .abr** ([docs/brushes.md](docs/brushes.md)): the active tip is deliberately NOT persisted (every launch resets to the Round startup preset).
- **Palette mode** ([docs/palette-mode.md](docs/palette-mode.md)): a per-document WRITE CONSTRAINT — pixels stay RGBA; never "upgrade" to index-per-pixel storage. Every NEW pixel-writing path must add the palette snap (six existing sites in the doc), and every palette mutation bumps `palette_revision`.
- **File formats & PSB & flat-image alpha** ([docs/file-formats.md](docs/file-formats.md)): new formats are one `file_format_entries()` row + one registry row + one writer branch; a layer-capable one must join `save_extension_preserves_layers()` or every save warns.
- **Camera raw** ([docs/file-formats.md](docs/file-formats.md) "Camera raw"): vendored LibRaw stock tarball ONLY (the separate demosaic packs are GPL — never vendor them); never byte-pin LibRaw output in tests.
- **HEIF/HEIC** ([docs/file-formats.md](docs/file-formats.md) "HEIF/HEIC"): read-only, PLATFORM codecs only (legal constraint above); tests are statistics-only and [SKIP] on known codec-unavailable messages.
- **Document channels** ([docs/channels.md](docs/channels.md)): saved alpha/spot planes are full-canvas `DocumentChannel`s, separate from the one raster mask per layer; merged `"Transparency"` is derived file plumbing, never a stored channel.
- **Resolution (PPI) & measurement units** ([docs/resolution-units.md](docs/resolution-units.md)): resolution is metadata, pixels stay the truth; untagged imports are 72 PPI via the density probe, NEVER Qt's screen-derived default.
- **PSD adjustment layers & clipping masks** ([docs/file-formats.md](docs/file-formats.md), [docs/ps-compat.md](docs/ps-compat.md)): every modeled kind writes a native Photoshop block — a kind without one breaks in Photoshop (a plAD-only Color Balance opened as an opaque white raster); `hue2` and `blnc` are patch-in-place (never regenerate hue2 band records or blnc shadows/highlights bytes); `plAD` is written only beside `levl`/`hue2` and its kind byte is FROZEN at the original four kinds; the Brightness/Contrast ADJUSTMENT formula is PS-legacy-calibrated and deliberately differs from the byte-pinned destructive filter (never unify either direction); Colorize uses measured PS tables (do not "simplify" to the ideal HSL ramp); sibling iteration goes through `composite_sibling_layers`, never a raw children loop.
- **Filter Gallery (visual filters), recipes, and Saved Looks** ([docs/filters.md](docs/filters.md)): the gallery's DISPLAY name is "Filter Gallery" (renamed July 2026) but every persisted identifier kept its original name (`filter.gallery`, `filters/gallery/*`, `filterGallery*` object names, file names). IDs, parameter keys, and captured colors are persistence contracts; unknown IDs/schemas make the whole recipe unsupported. Legacy `FilterRegistry::apply` and catalog defaults intentionally differ (separate canaries). Lens/Iris Blur are raster-only, fixed-geometry design-arounds; do not add native Blur Gallery authoring or broaden their algorithms.
- **Blend modes** ([docs/blend-modes.md](docs/blend-modes.md)): `BlendMode` is append-only; adding a mode touches blend_math, blend_mode_ui, the three PSD maps, and the Aseprite map — checklist and calibrated rounding rules in the doc.
- **Layer style presets & .asl** ([docs/style-presets.md](docs/style-presets.md)): built-in preset GUID ids and canonical names are persisted — append-only, never rename (pattern presets too); patchy_ja.ts is hand-maintained (bulk lupdate kills the dynamic preset-name entries).
- **Gradients & .grd** ([docs/gradients.md](docs/gradients.md)): `GradientDefinition` content is separate from `LayerStyleGradient` placement; preset selection must preserve placement. Default ids/names are append-only.
- **Layer styles** ([docs/ps-compat.md](docs/ps-compat.md)): untouched `lfx2`/`lrFX`/`lmfx` stays byte-preserved; edits regenerate PS 2026's captured descriptor shapes exactly. Multi-instance effects are read from 'lmfx' but always written inside lfx2 only. Group layer effects still do not render.
- **Text tool & Character panel** ([docs/text-tool.md](docs/text-tool.md)): Delete on a text layer deletes the OBJECT, never its pixels; any UI coexisting with an open editor session needs the `is_text_option_widget` exemption from focus-loss auto-commit.
- **Photoshop text model** ([docs/ps-compat.md](docs/ps-compat.md) "Photoshop text model"): imported type layers re-render via the `patchy.text.layout = photoshop` marker; native Patchy text KEEPS emitting runs v1/v2 + Qt-natural layout (the marker is the only opt-in — that keeps old files byte-stable).
- **Selection tools** ([docs/selection-tools.md](docs/selection-tools.md)): Quick Mask, Quick Select and Magnetic Lasso internals, `kSelectionToolCount` growth, and the marching-ants outline cache (direct `selection_` writes need `invalidate_selection_outline()`). The Quick Select patent rules above are binding.
- **Shape tools, Merge Down, tool icons** ([docs/tools.md](docs/tools.md)): shape options are session-only and ALL constraint math lives in `CanvasWidget::shape_drag_rect()`; merge_down flattens into the bottom-most visible item and hidden layers never block it.
- **Vector tools: pen paths, shape layers, vector masks, Paths panel** ([docs/vector-tools.md](docs/vector-tools.md)): shape layers stay `LayerKind::Pixel` with `patchy.vector.*` markers and structured `VectorShapeContent`/`LayerVectorMask` fields; pixels are a baked cache (never rasterize per repaint), PSD blocks follow the dirty-or-verbatim rule with encodings pinned by PS 27.8 observation, and `canvas_widget_pen.cpp` is TABLET input - the vector tools live in `canvas_widget_vector_tools.cpp`. Curvature Pen, content-aware tracing, align-edges auto-fitting, and on-canvas live-shape gizmos are patent-excluded pending review.
- **SVG import/export** ([docs/svg.md](docs/svg.md)): opens as editable shape layers and saves/exports vectors-preserved (deliberately NOT in `save_extension_preserves_layers` - warn + save-a-copy); the Qt-free reader hands text rendering and data-URI decoding to post-open UI passes via `patchy.svg.*` markers, and SVG is never byte-pinned against Photoshop (open W3C standard; fidelity is judged against independent renderers).
- **Single instance** (`src/app/main.cpp`): a per-user QLocalServer forwards file args (and `--screenshot`) to the running window; `PATCHY_NO_SINGLE_INSTANCE=1` opts out. Tests construct `MainWindow` directly, so they are unaffected.
- **Float windows** ([docs/float-windows.md](docs/float-windows.md)): `canvas_` is the single source of truth for the active document (never the tab widget's current index); every activation funnels through `activate_document_canvas`, close paths are session-based.
- **Import: scanner, sprite sheets, tile preview** ([docs/import.md](docs/import.md)): per-OS scanner backends with `PATCHY_FAKE_SCANNER_FILE` for offscreen tests; the macOS AppKit sheet must complete asynchronously.
- **Plugins** ([docs/plugins.md](docs/plugins.md)): the supported surface is the C ABI in `src/plugins/plugin_api.h`; plugins never store host pointers after a call returns. Legacy `.8bf` filters go through the adapter.
- **README screenshots**: generated, not hand-captured — `shot_readme_*` scenes in tests/ui/readme_screenshot_tests.cpp; regenerate with `scripts\make-readme-screenshots.ps1` (needs this machine's uncommitted `local-test-fixtures/`) and commit the PNGs. Keep `width="33%"` + `valign="top"` on every README screenshot `<td>`.
- **Performance / stress test** ([docs/performance.md](docs/performance.md)): the PATCHY 64 harness, `kStepBaselines` (step ids are persisted — never rename), smart invalidation, style-mask caching, parallel strip rendering.
- **Refactor backlog** ([docs/refactor-backlog.md](docs/refactor-backlog.md)): deferred cleanups from the July 2026 survey — test-suite split design, duplication inventory (what is safe vs canary-gated vs deliberate), megafunction dialog decomposition plans, lower-severity bug notes. Read it before starting cleanup or refactor work so findings are not re-derived.
