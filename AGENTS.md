## Build, test, and release handoff

For code changes, ALWAYS finish work in this repository by refreshing the local release build — `build\release\patchy.exe` must be freshly built from the final working tree at handoff, never stale. If the change is documentation-only or otherwise cannot affect compiled/runtime behavior (for example README text/images, comments, or agent instructions), do not run the full release build/test handoff just for that change; instead report that the release handoff was skipped because the change is non-code.

Required release handoff steps:

1. Build the release preset:

   ```powershell
   cmd /s /c '"C:\Program Files\Microsoft Visual Studio\18\Community\Common7\Tools\VsDevCmd.bat" -arch=x64 -host_arch=x64 >nul && "C:\Program Files\Microsoft Visual Studio\18\Community\Common7\IDE\CommonExtensions\Microsoft\CMake\CMake\bin\cmake.exe" --build --preset release'
   ```

   `build\release\CMakeCache.txt` carries hand-edited flags the preset does not set: `CMAKE_CXX_FLAGS_RELEASE=/O2 /Ob2 /DNDEBUG /Zi` and `CMAKE_EXE_LINKER_FLAGS_RELEASE=/DEBUG:FULL /INCREMENTAL:NO`. They make release builds emit `patchy.pdb`, which is what symbolizes user crash dumps (WER minidumps land in `%LOCALAPPDATA%\CrashDumps`). If you reconfigure from scratch, pass them again with `-D...`; to symbolize a dump from an older build, rebuild that commit in a temp `git worktree` with the same flags (full links reproduce the binary layout) and read the dump against that PDB.

   A running `build\release\patchy.exe` will lock the link step (`LNK1104`); ask Seth to close it rather than force-killing it (he may have unsaved work).

2. Run the release test binaries from `build\release`:

   ```powershell
   .\patchy_core_tests.exe
   $env:QT_QPA_PLATFORM='offscreen'; .\patchy_ui_visual_tests.exe
   ```

3. In the final response, explicitly report whether the release executable exists at:

   ```text
   build\release\patchy.exe
   ```

4. If the change touched platform-guarded code, CMakeLists/presets, or packaging, additionally run the affected remote build(s) best-effort — `scripts\remote\remote-build.ps1 -Target mac` and/or `-Target linux` — and report the result. A mac/linux failure does not block the handoff (Windows is the gate), but it must be reported, not silently skipped.

Do not say a release was created unless the release preset build completed successfully.

## Platform portability (Windows lead; macOS/Linux ports)

Windows is the lead platform and must never regress: the release handoff above stays mandatory for every code change. macOS and Linux build remotely via `scripts\remote\remote-build.ps1 -Target mac|linux` (snapshots the working tree including uncommitted changes — no commits, branches, or index changes — builds on the remote, runs both suites, streams output back). Changes that touch platform-guarded code, CMakeLists/presets, or packaging should run the affected remote build(s); pure Windows-only work need not. Remote-machine details, provisioning, and the full platform-specific site inventory live in [docs/platform.md](docs/platform.md).

- Default to a small, local `#ifdef Q_OS_WIN / Q_OS_MACOS / Q_OS_LINUX` (`_WIN32` in Qt-free code) **with a portable fallback**. Split into a per-OS translation unit (`foo_win.cpp` / `foo_mac.mm` / `foo_linux.cpp` behind `WIN32`/`APPLE`/`UNIX AND NOT APPLE` in CMakeLists, one shared header) only when a site needs Objective-C++/system frameworks or outgrows about a screenful; per-OS files live next to their feature, not in a platform/ directory.
- Window-frame code stays concentrated in `main_window_chrome.cpp`. `MainWindow::use_custom_window_chrome()` is the single gate: true only on Windows (frameless window + custom title-bar controls + Qt-level edge-resize machinery); macOS/Linux use the native frame, and on macOS the default QMenuBar becomes the global menu bar — never call `setNativeMenuBar(false)` outside the gated `configure_window_chrome()`.
- Tests obtain font files only through `tests/test_fonts.hpp` (role-based, per-OS candidate paths; the Windows lists are the exact historical ones so Windows baselines never move). Mac/Linux test-failure triage: fix real bugs > per-platform skip-with-reason > per-platform baseline; never loosen a tolerance globally to make another platform pass.
- File formats must stay byte-identical across platforms: PSD I/O is explicit big-endian fixed-width (`psd_binary.hpp`); keep any new serialization that way (no `size_t`/`long`/`wchar_t` writes, no raw struct dumps).

## MainWindow's implementation is split across main_window_*.cpp

`MainWindow` is still one class (`src/ui/main_window.hpp`), but its implementation spans several translation units. Put new member functions in the file that owns their area, and keep the split pure: moving a function between these files must never change its body.

- `main_window_chrome.cpp` — frameless-window machinery and `use_custom_window_chrome()` (the platform gate; deliberately concentrates the `Q_OS_WIN` window-frame code).
- `main_window_palette.cpp` — palette-mode document mutations, palette file I/O, panel/chip refresh, compliance scan.
- `main_window_adjustments.cpp` — Filter menu apply flow, adjustment dialogs, adjustment-layer create/preview/edit, async pixel-preview machinery.
- `main_window_shared.{hpp,cpp}` — helpers used by more than one of these TUs. Per-file helpers live in each TU's anonymous namespace; when a second TU needs one, MOVE it here and declare it in the header (never copy it — a duplicated helper with a static local forks its state, and an extern declaration alongside a same-name anonymous-namespace definition makes every call ambiguous). The split TUs repeat main_window.cpp's full include block; that is deliberate.

Do NOT attempt the text tool as a pure-move split: the text render pipeline is shared between too many members; it is really a "design a text_render module with its own header" job, not a file split (tried and backed out).

## Cross-cutting gotchas

- **Argument evaluation order: never read a variable beside `std::move(it)`.** `f(g(x), std::move(x))` is broken: C++ argument evaluation order is unspecified and MSVC goes right-to-left, so `x` is moved into the by-value parameter before `g(x)` reads it. This silently committed empty feathered selections. Compute `g(x)` into a local first, or use an overload that derives the value internally — `CanvasWidget::combine_selection_from_mask(bounds, mask)` exists for exactly this.
- **Reads must not bump layer revisions.** Layer's mutable accessors bump render/content revisions on ACCESS, so read-only code must go through const layers (`std::as_const`) or it silently invalidates every revision-keyed cache. Hunt regressions with `PATCHY_REV_TRACE=1` (stderr REVBUMP lines per accessor+layer).
- **Nothing O(layer pixels) may run per repaint.** Anything reachable from `paintEvent` must be cached or bounded. Use the layer's app-globally-unique content revision for whole-render results, or `pixel_revision()` for results that depend only on its pixel buffer; a full alpha-channel scan per frame made zoom/pan take ~500 ms on a 70 Mpx layer. See [docs/performance.md](docs/performance.md).
- **Cross-toolchain determinism in core algorithms.** RNG is splitmix64 with explicit uniform mapping — NEVER `std::uniform_*_distribution` (implementation-defined output breaks cross-toolchain pixel tests). Geometry/graph code uses integer math or deterministic double envelopes with fixed tie-breaks (brush_dynamics, magnetic_lasso, the EDT in layer styles all follow this).
- **Read folded event modifiers, not `QApplication::keyboardModifiers()`.** The live state can lag the app-level event filter (stuck eyedropper cursor) and offscreen tests never clear it (see Testing notes). Drive cursor/mode changes from the event's modifiers.
- **Append-only enums**: `BlendMode` (core/layer.hpp) and `BrushDynamicControl` ride combo item data, casts, and file maps keyed on the existing order — only append.
- **Never rename persisted identifiers**: hotkey command ids, palette preset ids, stress-test step ids, and settings keys are all stored in user settings or files.
- **Byte-stable serialization canaries**: `psd_layered_writer_bytes_are_stable`, `gif_encoder_bytes_are_stable`, and `tool_write_paths_digest_baseline` pin default-path output byte-identically (FNV hashes). Re-pin only for deliberate format/rendering changes, never to make a refactor pass.
- **The CPU compositor is the reference implementation**: any GPU or optimized rendering path must match its golden output (the pinned compositor tests and canaries encode this).
- **Chrome dialogs that react to closing must funnel through `done()`.** `reject()` (chrome X, Esc) hides WITHOUT a QCloseEvent, so closeEvent-based cleanup misses it; but overriding `reject()` to call `close()` is worse: `QDialog::closeEvent` itself calls `reject()` and treats a still-visible dialog as a vetoed close, turning EVERY close path into a no-op (pinned by `ui_tile_preview_window_tracks_document_edits`).
- **New non-modal dialogs must go through `run_non_modal_dialog`** (dialog_utils): it auto-rejects child dialogs when their parent finishes and applies the macOS above-parent native anchoring. Bypassing it reintroduces the buried/orphaned-dialog bugs. `request_patchy_color` allows one picker at a time; transient pickers keep their own position-memory group.
- **QListWidget rows built with `setItemWidget` must paint their own selection** — the opaque row widget hides the `::item:selected` QSS. Give such lists `padding: 0` on `::item` (padding insets the item-widget geometry and bleeds rows), set `background: transparent` on the row's QLabel/QCheckBox children, and keep the app stylesheet's global `QCheckBox { border: none; }` (suppresses QMacStyle's Aqua layout-item margins; removing it jams labels into checkboxes on mac, and offscreen tests cannot catch it).
- **Options-bar controls share one fixed 26px row height** (app stylesheet); a taller control grows the toolbar only while its tool is active and the canvas shifts on tool switch. QStyleSheetStyle inflates QToolButton size hints by +3px, so bar QToolButtons need an explicit QSS min/max-height cap. A free-transform/warp session owns the bar (hides per-tool widgets) to keep the single row.
- **Application-wide options-bar state must follow the `current_*` mirror pattern** in MainWindow: record the value in a member on the control's signal, apply it at BOTH per-session sites (the new-session setup block and `activate_document_tab`), and hand-update the mirror in `load_tool_settings()` under `QSignalBlocker`. A setter-only-on-current-canvas lambda silently desyncs documents created/activated later (the shape Fill checkbox bug; `ui_shape_fill_and_corner_radius_apply_to_new_documents`).
- **Status bar widgets**: QStatusBar hides every non-permanent widget while a message shows (Patchy almost always shows one), so left-side widgets are manually positioned children of the `ZoomStatusBar` subclass (`src/ui/zoom_status_bar.*`), never `addWidget()`.
- **Direct writes to `selection_` / `selection_display_region_`** (not via `set_selection_from_region` / `set_selection_from_mask` / `restore_selection_before_edit` / `apply_selection_snapshot` / `reselect`) must call `invalidate_selection_outline()` or the marching ants render stale.
- **`kSelectionToolCount`** growth means updating BOTH brace-initializers (canvas_widget.hpp `selection_modes_per_tool_`, main_window.hpp `selection_modes_`) plus the tools array in `apply_selection_modes_to_canvas`.
- **App-level shortcuts vs canvas keys**: while a magnetic-lasso trace is live, `CanvasWidget::event()` accepts the ShortcutOverride for Delete/Backspace so the `layer.clear` shortcut never consumes them; use the same pattern for any tool that needs keys an app shortcut owns.
- **CMake runtime assets are shared copy-once targets** (`patchy_bundled_fonts`, `patchy_qt_runtime`, `patchy_qt_base_translations`). Never attach per-target POST_BUILD copies into the shared output directory — concurrent copies race under parallel Ninja (intermittent release-build failures). New executables call the existing `patchy_copy_*` helpers.
- **`GradientStopsEditorWidget` is callback-driven**: it never mutates its own stop vectors; hosts copy-and-sort, and must never sort the working vectors in place - an in-flight tag or midpoint drag holds an index into them. Photoshop `Mdpn` belongs to the destination/right stop; the first stop's midpoint is unused.
- **Adding a brush-dynamics field means updating BOTH equality functions**: `brush_dynamics_is_default` AND `persisted_dynamics_equal` — missing one silently breaks sidecar writes or factory reset.
- **Dialog spin boxes that keep their -/+ buttons** must append `dialog_spinbox_button_style()` (src/ui/dialog_utils) to the dialog stylesheet AFTER all children exist (QSS sub-control gotcha documented in that header).
- **Layer context menu: Edit Layer Styles... stays the FIRST item, always** (Seth's standing rule; `ui_layer_context_menu_keeps_edit_styles_on_top` pins it).
- **NEVER SAVE OVER the PSBtest tent/Content files** (`local-test-fixtures/psd/PSBtest/`): several `_if_available` tests byte-pin their PS-2023 shapes, and a Patchy save re-stamps them. Pristine originals live in `D:\projects\C2\TableTents\DrinkTable\` — re-copy them if the suite's liFE trailer checks start failing.

## Legal constraints (patents and licensing)

- **Quick Select solves ONCE on mouse-release; do not add live per-mouse-move classification or selection preview before Nov 3, 2029.** Adobe's US 8050498 ("Live coherent image selection", term-adjusted) claims classify-and-display while brush input is being received. Also deliberate: Enhance Edge is purely geometric majority smoothing (US 8013870 claims local-color-model edge opacity until 2028), the solve uses ONE window per stroke (US 8121407 claims overlapping-tile decomposition until 2030), and there is no input-driven auto-switching between segmentation algorithms (US 10698588). `ui_quick_select_stroke_selects_object_and_is_undoable` asserts the no-mid-drag behavior. Details and the expired foundations in [docs/selection-tools.md](docs/selection-tools.md).
- **Magnetic Lasso patents are all expired** (US 5,995,115 expired 2017-04-04; the technique is 1995-1997 published prior art), so live snap DISPLAY is fine there — but the selection REGION is still built once, in `finish_magnetic_lasso()`, staying clearly outside US 8050498.
- **Brush shape dynamics/scattering** (Photoshop 7, 2002): covering patents expired; no design constraint.
- **Tool icons are original SVGs** (`src/ui/icons/tool-*.svg`): generic metaphors, no Photoshop geometry, so the license stays clean. Palette preset provenance lives in NOTICE-THIRD-PARTY.md.
- **Bundled art and textures must never come from AI image generation** (unclear provenance, plus upscaling/JPEG slop). The two acceptable sources are code generation (the brush-tip / pattern-preset model: deterministic, no binary assets) and real photographs or human-artist work under a commercial-compatible license (CC0 strongly preferred; avoid attribution-required licenses for bundled assets when possible). Every bundled third-party asset gets a NOTICE-THIRD-PARTY.md entry with source URL, author, and license, and ships lossless (process from PNG masters; never re-encode through JPEG). When picking from ambientCG, filter by the API's per-asset `creationMethod` (photogrammetry/photo only); Poly Haven textures are human-made per their license page.
- **HEIC/HEIF decode comes from PLATFORM codecs only — never vendor libheif, libde265, x265, or any other HEVC decoder/encoder.** HEVC is heavily patent-encumbered (Access Advance/Via LA pools plus bilateral holders like Nokia, who also hold HEIF container patents to ~2035); using WIC on Windows, Qt's ImageIO-backed qmacheif on macOS, and the KDE runtime + ffmpeg-full extension on Linux keeps every decoder someone else's licensed product. The GIMP/Krita bundle-libde265 posture was researched and deliberately rejected (July 2026, decision by Seth). Encoding HEIC is off the table entirely. Details in [docs/file-formats.md](docs/file-formats.md).
- **Dependency licensing**: commercial-friendly dependencies only in shipping binaries. GPL tools may be used for local development or test generation, but only when they are not linked or distributed.

No Photoshop SDK code or Adobe-created assets are ever allowed in this repository or in any Patchy binary. The only exception is the `test-fixtures/psd/` regression files, which are all authored by Seth and/or the Patchy team, and are not Adobe assets.

## Testing notes

- `patchy_ui_visual_tests.exe` must run with `QT_QPA_PLATFORM=offscreen`. The offscreen platform does **not** enumerate installed Windows fonts; register what a test needs with `QFontDatabase::addApplicationFont("C:/Windows/Fonts/<file>.ttf")`. Never call `removeApplicationFont` to clean up — invalidating an in-use font cache can hard-crash the suite.
- A registered font may not appear under its familiar GDI name: the offscreen FreeType database uses the OpenType *typographic* family, so ariblk.ttf registers as family "Arial" + style "Black", not "Arial Black". Patchy's text code resolves such names via `available_text_family_style_match` (main_window.cpp); tests should not gate on `QFontDatabase::families().contains("Arial Black")`-style checks.
- The test harness `CHECK()` macro throws. A failing CHECK while a `MainWindow` with an open inline text editor is still alive aborts the process during unwind (exit code 3, no `[FAIL]` line printed). Prefer asserting after the editor session is committed/closed.
- Tests save PNGs via `save_widget_artifact(...)` into `test-artifacts/` next to the binary — inspect them to visually confirm rendering behavior.
- The offscreen platform does **not** clear `QApplication::keyboardModifiers()` after a synthetic `QKeyEvent` (a release never unsets the bit), and the bit persists across tests in the shared `QApplication`. Assert cursor/mode changes through code paths that read the event's *folded* modifiers (e.g. `CanvasWidget::eventFilter`), not `keyboardModifiers()`. See `ui_brush_alt_shows_eyedropper_cursor` for the order-independent pattern.
- A click on a layer-row mask/content thumbnail can rebuild the layer row (the old row widget is deleted), so never reuse a cached row/thumbnail pointer across the press — use `click_layer_row_thumbnail(...)` in ui_visual_tests.cpp, which refetches the widget for press and release. Reusing the stale pointer is a use-after-free that flakes only when the heap reuses the freed block.
- If the visual suite dies with an access violation, the log ends with a symbolized stack (a dbghelp vectored handler in `main`) — read it instead of re-running and hoping.
- Run a subset of visual tests by passing a name substring as the first argument (or `PATCHY_UI_TEST_FILTER`): `.\patchy_ui_visual_tests.exe ui_audio_splitter`. There is no `--test` flag.
- Tests that open notice-raising files assert `statusBar()->currentMessage()` (import notices go to the status bar by default); only tests that ENABLE `imports/showPsdWarningsAndInfo` need the REPEATING QTimer dismisser — a one-shot fires during the open-progress phase and the suite hangs (see `ui_import_notices_dialog_shown_when_setting_enabled`).
- Some tests skip per platform by design (Windows-only chrome/plugins, font metrics); the current list with reasons is in [docs/platform.md](docs/platform.md). Local-fixture (`local-test-fixtures/`) tests `[SKIP]` on the remote machines because that directory is deliberately untracked.

## Driving and inspecting the app

- **Native Windows visual QA uses Patchy's own screenshot command, never Computer Use or other desktop-control automation.** Launch the release app and capture its real native styling with `patchy.exe --screenshot`; inspect the resulting image directly.
- `patchy.exe --screenshot <out.png>` saves a grab of the RUNNING instance's window without raising or focusing it (add `--screenshot-widget <qtObjectName>`, e.g. `layerList`, and/or `--screenshot-rect x,y,w,h` for a crop). The invoking process exits immediately; poll for the output file. Combine with positional files to open a document first. With no instance running, the new process shows the window, captures ~1.5s after startup, and exits (0 saved / 3 failed). Coverage: `ui_debug_screenshot_saves_window_widget_and_region`.
- `patchy.exe --stress-test[=quick|small|standard|huge] [--stress-report-dir <dir>]` runs the built-in performance scene and exits; reports land in `%APPDATA%\Patchy\stress-reports\` (read `stress-latest.json`). Meaningful numbers need a REAL screen; quick (1024 px) for iteration, standard (4096 px) for full-scale. Details in [docs/performance.md](docs/performance.md).
- Debug env vars: `PATCHY_NO_SINGLE_INSTANCE=1` (allow multiple instances), `PATCHY_FAKE_SCANNER_FILE=<path>` (bypass WIA COM for scanner-import tests), `PATCHY_REV_TRACE=1` (log revision bumps), `PATCHY_ZOOM_TRACE=1` (paint/zoom phase timings over 2 ms), `PATCHY_STYLE_MASK_CACHE_OFF=1`, `PATCHY_RENDER_SINGLE_THREADED=1` (byte-stable sequential render for cross-run checksum compares), `PATCHY_PROCESSING_OVERLAY_MIN_PIXELS` (overlay threshold override), `PATCHY_UI_TEST_FILTER` (test subset).
- Composite checksums (stress report FNV-1a and any big-render comparison) are comparable on one machine only: text AA varies across machines and the parallel strip renderer makes them thread-count dependent.

## Reviewing a contributor PR ("let's look at this PR…")

When Seth points at a PR and wants it reviewed, do these steps in order and let him drive the merge decision:

1. **Read it first.** `gh pr view <N>` / `gh pr diff <N>`, review in context (open touched files, verify referenced symbols exist). Explicitly check for malicious or sketchy code — network calls, filesystem/process/shell access, new dependencies, obfuscation. Report a short verdict.
2. **Build & test it locally — never create remote branches or worktrees.** Fetch the head into a local branch (`git fetch origin pull/<N>/head:pr-<N>`), check it out, run the release build, then the relevant test subset plus the full UI + core suites.
3. **Let Seth test the built `patchy.exe` himself** and wait for his go-ahead. Do **not** merge until he says to.
4. **Confirm the merge path before merging** — straight `gh pr merge <N> --merge`, or a local `git merge --no-ff pr-<N>` on main when follow-up commits (e.g. missing translations) should ride along, then push; GitHub marks the PR merged once its head is reachable from main. Never squash contributor PRs — it would strip their commit attribution.

## Photoshop compatibility verification

Adobe Photoshop 2026 is installed on this machine and is the ground truth for PSD compatibility work. It is COM-scriptable from PowerShell: `(New-Object -ComObject Photoshop.Application).DoJavaScript($jsx)` (the first call launches Photoshop, ~30s).

- To learn how Photoshop encodes a setting, save two PSDs differing in exactly one UI toggle and byte-diff them. To check how PS interprets a Patchy-written file, query Action Manager getters (`executeActionGet`) or diff PS's flattened export against `Compositor::flatten_rgb8`.
- Committed PS-authored regression fixtures live under `test-fixtures/psd/`; generate new ones with a COM script rather than hand-crafting bytes.
- Two silent-corruption write rules: layer record flags **bit 3** must be written on every layer (without it PS applies legacy semantics and misrenders unlinked-mask + effects combinations), and lfx2 effect **blend modes must be written as full stringIDs** ("multiply", not 'Mltp') — PS silently reads 4-char codes as Normal. The GrFl (gradient overlay) descriptor is additionally shape-sensitive (must mirror PS's 14-item layout).
- COM techniques, script hygiene, per-effect calibration notes (Stroke band, Spread, Choke, glow falloff), and the scratch-tool compile line are in [docs/ps-compat.md](docs/ps-compat.md).

## Feature index

Read the linked doc before working on a feature; each entry below carries only the constraint most likely to bite.

- **Smart objects** ([docs/smart-objects.md](docs/smart-objects.md)): a smart object stays a `LayerKind::Pixel` layer (the text-layer pattern) identified by `patchy.smart_object.*` metadata (core/smart_object.hpp); untouched layers re-emit their SoLd/PlLd blocks byte-identically, edited ones regenerate patch-in-place via psd_descriptor.cpp. E1-E14 refer to the PS 2026 COM ground-truth captures documented there.
- **Warp** ([docs/warp.md](docs/warp.md)): warped smart objects, the 15 warp-style mesh constructions, the Warp Transform tool, and Warp Text all share `generate_style_warp_mesh`/`apply_warp_distortion` (core/warp_mesh). Meshes regenerate from parameters (never stored in Patchy metadata beyond the file's own descriptor); box text warps over the dragged FRAME — that IS Photoshop's behavior, guarded by `ui_warp_text_box_text_warps_over_frame`.
- **Brush tips & dynamics & .abr** ([docs/brushes.md](docs/brushes.md)): the active tip is deliberately NOT persisted (every launch resets to the Round startup preset); the old `tools/brushTip`-era settings keys are neither read nor written.
- **Palette mode** ([docs/palette-mode.md](docs/palette-mode.md)): a per-document WRITE CONSTRAINT — pixels stay RGBA; never "upgrade" to index-per-pixel storage. Every palette mutation bumps `palette_revision` via `MainWindow::next_palette_revision()` (never reused), and every NEW pixel-writing path must add the same palette snap (five existing sites: `write_pixel`, `flood_fill`, `smudge_brush_segment`, `write_brush_stroke_pixel_from_snapshot`, `clone_brush_segment`).
- **File formats & PSB & flat-image alpha** ([docs/file-formats.md](docs/file-formats.md)): new formats slot in with one `file_format_entries()` row + one registry row + one writer branch. Writing >30k px as `.psd` errors ("use .psb", cap 300k). Non-PSD flat alpha remains a marked editable layer mask; PSD saved alpha never uses the old `"Alpha 1"` name heuristic. Saving a layered document to a flat format is guarded Photoshop-style (Save routes to Save As defaulting to .psd; a confirmed flat save is a copy that leaves the session modified) - a new layer-capable format must join `save_extension_preserves_layers()` or every save warns.
- **Camera raw** ([docs/file-formats.md](docs/file-formats.md) "Camera raw" section): vendored LibRaw (CDDL election; stock tarball ONLY — the separate demosaic packs are GPL and must never be vendored) behind Qt-free `formats/raw_document_io`; read-only sources (Save routes to Save As .psd), develop dialog gated by `imports/showRawDevelopDialog`, `imports/rawDevelop*` settings keys are persisted contracts. Never byte-pin LibRaw output in tests (float pipeline varies per toolchain).
- **HEIF/HEIC** ([docs/file-formats.md](docs/file-formats.md) "HEIF/HEIC" section): read-only, PLATFORM codecs only (WIC / qmacheif / KDE runtime + ffmpeg-full Flatpak extension — see the legal constraint above; never vendor an HEVC codec). Windows codec-missing errors carry marker prefixes that become an "Open Microsoft Store" button; tests are statistics-only and [SKIP] on the known codec-unavailable messages.
- **Document channels** ([docs/channels.md](docs/channels.md)): saved alpha/spot planes are full-canvas `DocumentChannel`s, separate from the one raster mask on each layer. Merged `"Transparency"` is derived file plumbing, never a stored channel; PSD resource 1053 enumerates alpha identifiers only and skips spot channels.
- **PSD adjustment layers & clipping masks** ([docs/file-formats.md](docs/file-formats.md), [docs/ps-compat.md](docs/ps-compat.md)): Levels + Curves + Hue/Saturation are native PSD (`levl`/`curv`/`hue2`). `hue2` is patch-in-place; never regenerate its band records. Untouched `curv` bytes are exact-preserved and edited curves regenerate the PS2026 v1-bitmap + indexed-v4 shape; `.acv` shares the curve-body parser. Never write `plAD` beside editable `curv`: Photoshop warns about unknown data; legacy private Curves migrates to native on save. Colorize renders through measured PS tables in core/adjustment_layer.cpp (do not "simplify" them to the ideal HSL ramp). Clipping groups composite via `composite_sibling_layers` - every sibling-iteration site must go through it, never a raw children loop.
- **Visual filters, recipes, and Saved Looks** ([docs/filters.md](docs/filters.md)): IDs, parameter keys, categories, scaling, and captured colors are persistence contracts; unknown IDs/schemas make the whole recipe unsupported. User Looks are independent UUID-named v1 JSON records and structurally valid unsupported recipes stay preserved. Legacy `FilterRegistry::apply(id, pixels)` and named catalog defaults intentionally differ for several filters and have separate output canaries - never redirect or normalize one into the other.
- **Blend modes**: `BlendMode` is append-only (see gotchas). Adding a mode means updating blend_math.cpp, blend_mode_ui.cpp (display order decoupled via item data), the three PSD maps (4-char key + lfx2 stringID read AND write), and the Aseprite map in both directions. Non-separable modes use the PDF-spec set_lum/set_sat algorithm; Exclusion rounds the s*d/255 product BEFORE doubling and Divide rounds to nearest (verified against both Photoshop and Aseprite; `aseprite_blend_modes_match_aseprite_render` pins the latter in-suite).
- **Layer style presets & .asl** ([docs/style-presets.md](docs/style-presets.md)): the dialog's Style Presets page + Style Manager sit on a folder-aware `StyleLibrary` (one self-contained single-style .asl per entry) with 39 built-in presets whose GUID ids and canonical English names are persisted — append-only, never rename; per-entry `introduced_version` + the version-2 defaults gates seed only new entries on upgrade. 20 bundled CC0 photo-texture patterns (Poly Haven, "Textures" folder, qrc-embedded — a static-lib qrc needs the Q_INIT_RESOURCE force-registration documented in the doc) back the Materials styles; resolve bundled ids only via ui/photo_pattern_presets.hpp's helpers. The .asl codec shares the lfx2 conversion (psd/psd_layer_effects.hpp); `asl_writer_bytes_are_stable` and `ui_photo_pattern_presets_load_stable_tiles` pin bytes, and patchy_ja.ts is hand-maintained (bulk lupdate kills the dynamic preset-name entries — see the doc).
- **Gradients & .grd** ([docs/gradients.md](docs/gradients.md)): reusable `GradientDefinition` content is separate from `LayerStyleGradient` placement; preset selection must preserve placement, and layer styles resolve dynamic FG/BG roles before storage. The 20 default ids/names are append-only. GRD v5 hierarchy is a trailing `8BIMphry` descriptor.
- **Layer styles** ([docs/ps-compat.md](docs/ps-compat.md)): untouched `lfx2`/`lrFX` stays byte-preserved. Edited Satin writes Photoshop's 12-field `ChFX` Linear-contour shape; custom SATIN contour/`AntA` stays raw until edited. Pattern Overlay and the Bevel Contour/Texture sub-options are fully modeled: pattern pixels ride the document-global `Patt` blocks (imported blocks stay raw-preserved AND decode into `DocumentMetadata::patterns`; Patchy appends its own `Patt` block only for referenced uncovered ids), bevel contour points round-trip exactly, and the 12 built-in pattern presets (src/core/pattern_presets.cpp) have PERSISTED GUID ids — never rename or re-seed them (`pattern_presets_generate_stable_tiles` pins the tiles). Standalone Photoshop `.pat` presets live in the folder-aware user `PatternLibrary`; selecting one copies it into the document, so library deletion must never mutate an open document. Same-id/different-pixel picks receive a document-local UUID, and layer-style preview/cancel must restore the original pattern store before the undo snapshot. The regenerated `ebbl`/`patternFill` descriptors mirror PS 2026's captured layouts (conditional sub-option keys omitted while off). RGB Blend If uses the native 40-byte Gray/R/G/B range payload; unknown/color-mode shapes and group-boundary ranges remain preservation-only. PS stores multi-instance effects in 'lmfx' (same payload shape as lfx2): Patchy reads it as authoritative over the compatibility lfx2 and drops it on edit, but always writes multi lists inside lfx2 only. Group layer effects still do not render.
- **Text tool**: a Type-tool click inserts a provisional 1x1 text layer immediately (marker `patchy.internal.provisional_text`); `commit_text_editor` removes it first via the marker-checked `MainWindow::take_provisional_text_layer` (a stale id can never delete an unrelated layer), then snapshots and recreates the committed layer under the same id — cancel/empty-commit leaves history and modified state untouched. Delete on a text layer deletes the OBJECT, never its pixels: pixel-clearing leaves an invisible layer whose metadata resurrects the text (`clear_active_layer` special-cases it; mixed selections clear pixels + delete text layers in one undo step). Mutating actions that take no focus (e.g. the layer lock buttons) must call `finish_active_text_editor()` first. The options bar shows session apply/cancel buttons (`textApplyButton`/`textCancelButton`) while an editor is open; they must keep `Qt::NoFocus`, otherwise the editor's focus-loss auto-commit fires on mouse press and Cancel commits instead of canceling. The font combo is a `FontPickerCombo` (src/ui/font_picker.*, a QFontComboBox whose overridden showPopup opens a searchable list + writing-system preview); its popup objectName `textFontPickerPopup` must stay matched by `is_text_option_widget` (a Qt::Popup is a window, so isAncestorOf-based ownership misses it and focusing the search box would auto-commit the session).
- **Photoshop text model** ([docs/ps-compat.md](docs/ps-compat.md) "Photoshop text model"): PSD-imported type layers carry `patchy.text.layout = photoshop` and re-render with Photoshop's COM-calibrated leading rules (per-line max leading, auto = 1.2 x size, engine units = document px, transform y-scale folds into sizes at render, box first baseline = sTypoAscender). Runs metadata v3 (double sizes, `auto` leading column, tracking, H/V glyph scales) — readers must keep accepting v1/v2, and native Patchy text must KEEP emitting v1/v2 + Qt-natural layout (the marker is the only opt-in; that's what keeps old files byte-stable). Missing per-run properties fall back to the engine data's normal style sheet, never to a sibling run's value.
- **Character panel** (options bar > Character... while the Text tool is active): edits the LIVE editor session (leading auto/fixed, tracking, H/V glyph scales) per selection. Its dialog (`textCharacterDialog`) is exempted from the editor's focus-loss auto-commit via `is_text_option_widget` — any new UI that must coexist with an open text session needs the same exemption. Setting fixed leading opts the layer into the Photoshop layout marker at commit (explicit leading does not render under Qt-natural layout).
- **Selection tools** ([docs/selection-tools.md](docs/selection-tools.md)): Quick Mask temporary selection-alpha editing, Quick Select and Magnetic Lasso engine internals, and the traced/cached marching-ants outline. Quick Mask stays canvas-owned and non-serialized; the patent rules above remain binding for Quick Select.
- **Shape tools**: rectangle/ellipse honor brush Opacity/Soft through the signed-distance renderer in pixel_tools.cpp; options-bar Style/Width/Height are deliberately session-only (mirrored via `current_shape_*`, never persisted). Alt = draw-from-center, so Rectangle/Ellipse are exempt from the Alt temporary-eyedropper (the two meanings fight). ALL constraint math lives in `CanvasWidget::shape_drag_rect()` — it intentionally duplicates `marquee_selection_rect()` (selection-only concerns, separately pinned). The Fill command has its OWN persisted Opacity/Soft (`tools/fillOpacity`/`tools/fillSoftness`, default 100/0); tests laying down setup color call `use_solid_fill_settings(canvas)`.
- **Merge Down** (`MainWindow::merge_down()`): flattens folders and any multi-selection into the bottom-most visible item, Photoshop-style. Hidden layers are discarded, never blocking (they must not abort the merge); cross-folder leaf merges may leave an emptied folder behind — intended. Coverage: `ui_merge_down_*`.
- **Single instance** (`src/app/main.cpp`): a per-user QLocalServer forwards file args (and the `--screenshot` command above) to the running window; `PATCHY_NO_SINGLE_INSTANCE=1` opts out. Tests construct `MainWindow` directly, so they are unaffected.
- **Float windows** ([docs/float-windows.md](docs/float-windows.md)): Window > Float in Window moves a document's canvas into a MainWindow-parented top-level (`DocumentFloatWindow`) so two documents are visible at once. `canvas_` is the single source of truth for the active document (never derive it from the tab widget's current index): every activation source funnels through `activate_document_canvas`, canvas history callbacks route to `session_for_canvas(canvas)` at fire time, and close paths are session-based (`close_document_session`), never tab-index loops.
- **Import menu / sprite sheets / tile preview**: scanner import is Windows-only WIA (`scanner_import_win.cpp`; `PATCHY_FAKE_SCANNER_FILE` for offscreen tests); sprite sheets are the testable `compose_sprite_sheet`/`slice_sprite_sheet` pair; the Seamless Tile Preview is a Qt::Tool window polling a CONST-only revision probe (mutable Layer accessors bump revisions — see gotchas) and closes via a `done(int)` override (see the chrome-dialog gotcha).
- **Plugins**: the supported surface is the C ABI in `src/plugins/plugin_api.h` (`PATCHY_PLUGIN_ABI_VERSION` must match the host; identifiers are reverse-DNS; host-owned memory stays host-owned, so plugins must never store raw document pointers after a call returns). Legacy Photoshop `.8bf` filters go through `src/plugins/legacy_photoshop_adapter.*` (inherently OS/architecture-matched; the Windows-only bundled shims are exercised by `ui_bundled_legacy_plugin_action_applies_filter`).
- **Tool palette icons**: original SVGs at `src/ui/icons/tool-*.svg` (32x32, `#dce2eb` strokes, at most one non-load-bearing `#74c0ff` accent). QtSvg renders a Tiny-1.2 subset only, and a typo'd qrc alias renders EMPTY silently — review with `patchy_ui_visual_tests.exe ui_tool_palette_icons` (writes contact-sheet artifacts, checks per-icon coverage + pairwise distinctness).
- **README screenshots**: generated, not hand-captured — `shot_readme_*` scenes in ui_visual_tests.cpp; regenerate with `scripts\make-readme-screenshots.ps1` (needs this machine's uncommitted `local-test-fixtures/`; commit the updated PNGs). Keep `width="33%"` + `valign="top"` on every README screenshot `<td>` or GitHub mis-sizes/mis-aligns the gallery.
- **Performance / stress test** ([docs/performance.md](docs/performance.md)): the PATCHY 64 harness, baselines (`kStepBaselines`, step ids are persisted — never rename), smart invalidation, style-mask caching, parallel strip rendering, deferred full-refresh paint.
